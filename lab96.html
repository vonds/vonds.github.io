Lab 96: Running Containers with Podman

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Run containers with Podman using a rootless workflow, then inspect, restart, log, and clean up containers and images. Practice basic volume mounts so you can persist data between container runs.

containers podman rootless rhel

Scenario

You’re preparing a host to run containerized utilities and small services using Podman. Your job is to install Podman, pull an image, run a named container, inspect its state, view logs, practice a bind mount for persistence, and then clean up containers and images in a controlled way.

Operator context

Rootless Podman is a common default on modern systems. The goal is to build the habit of verifying state (ps, inspect, logs) before you destroy or replace anything.

Objective

  • Install Podman and confirm it runs as a non-root user.
  • Pull an image and run a named interactive container.
  • Verify stopped/running state and restart by name.
  • Read container logs and inspect container JSON.
  • Run a container with a bind mount for persistent data.
  • Remove containers and images cleanly.

What You’ll Practice

  • Podman image lifecycle: pull, images, rmi.
  • Container lifecycle: run, ps -a, start, logs, inspect, rm.
  • Bind mounts: -v HOST:CONTAINER to persist data.
  • Safer cleanup patterns (remove a target container/image first, then do broader cleanup when you mean it).

Walkthrough

Step 1 : Install Podman.
Command 
# Debian/Ubuntu
sudo apt install podman -y

# RHEL/Fedora
sudo dnf install podman -y

# Arch
sudo pacman -S podman

After installation, your default workflow should be to run Podman as your normal user unless you have a specific reason to escalate.

Step 2 : Confirm Podman is installed and usable as non-root.
Command 
podman --version

This confirms the binary is available and helps you sanity-check environment issues before troubleshooting deeper.

Step 3 : Pull a base image.
Command 
podman pull alpine

Keep the mental model: an image is a template; a container is a running (or stopped) instance created from that template.

Step 4 : Run a named interactive container.
Command 
podman run --name lab-alpine -it alpine /bin/sh

You can exit the shell with exit or Ctrl+D. After that, verify state from the host.

Step 5 : Verify the container exists and is stopped.
Command 
podman ps -a

podman ps only shows running containers. Add -a when you want the full picture.

Step 6 : Start the container by name.
Command 
podman start lab-alpine

If you want to attach interactively after starting, you can use podman attach lab-alpine.

Step 7 : View logs for the container.
Command 
podman logs lab-alpine

Logs are often the fastest signal for what the container actually did when it started.

Step 8 : Practice a bind mount.
Command 
# Create a host directory first (so the mount is predictable)
mkdir -p ~/data

# Run an interactive container with a host directory mounted at /data
podman run -v ~/data:/data -it alpine /bin/sh

Anything written under /data inside the container lands in ~/data on the host.

Step 9 : Inspect container metadata in JSON.
Command 
podman inspect lab-alpine

This is the “source of truth” view for mounts, networking, entrypoint, environment variables, and runtime configuration.

Step 10 : Clean up containers and images.
Command 
# Remove the named container explicitly first
podman rm -f lab-alpine

# If you created additional containers during mounts, remove them too:
podman rm -a

# Remove the alpine image (or all images if you truly intend to wipe)
podman rmi alpine
# OR (broad wipe)
podman rmi -a

Prefer targeted removal in real environments. Use “remove all” commands when you’re sure the host is dedicated to labs.

Reference

  • Images
    • podman pull IMAGE
    • podman images
    • podman rmi IMAGE
  • Containers
    • podman run ...
    • podman ps / podman ps -a
    • podman start NAME
    • podman logs NAME
    • podman inspect NAME
    • podman rm NAME / podman rm -a
  • Volumes / Mounts
    • -v HOST_DIR:CONTAINER_DIR
