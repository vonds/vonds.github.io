Lab 109: Persistent Module Configuration

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Persist kernel module behavior across reboots by blacklisting an unwanted module and setting module options under /etc/modprobe.d. Validate that your configuration is applied and rebuild initramfs so early boot honors the policy.

kernel modules hardening

Scenario

A workstation fleet is loading an unnecessary legacy module that creates noise and confusion during troubleshooting. You will enforce a persistent blacklist for an unwanted module and define persistent options for a safe practice module, then validate the policy using standard tooling.

Operator context

Persistent module policy should be documented and reviewed. A blacklist can disable hardware functionality, and module options can change system behavior in ways that are not obvious during incident response.

Objective

  • Create a persistent blacklist entry for an unwanted module using /etc/modprobe.d.
  • Verify that the blacklist is present and effective.
  • Create a persistent module option configuration for a safe module.
  • Validate that module configuration is recognized by modprobe.
  • Rebuild initramfs so early boot honors module policy.
  • Confirm expected load state using lsmod and a test load operation.

What Youâ€™ll Practice

  • Persistent module policy using /etc/modprobe.d/*.conf .
  • Blacklisting behavior and verification methods.
  • Defining module options and validating effective configuration.
  • Rebuilding initramfs using common distro tooling (update-initramfs, dracut, mkinitcpio ).
  • Load-state verification using lsmod and /proc/modules .

Walkthrough

Step 1 : Persistently blacklist pcspkr.
Command 
echo 'blacklist pcspkr' | sudo tee /etc/modprobe.d/blacklist-pcspkr.conf

Files under /etc/modprobe.d are read by the module loader. Blacklisting prevents automatic loads and typically blocks manual loads as well, depending on distro policy and tooling behavior. 

blacklist pcspkr
Step 2 : Verify the blacklist entry exists.
Command 
grep -R '^blacklist pcspkr' /etc/modprobe.d

This confirms the policy is written to disk in the correct location. If your environment has multiple blacklist files, this helps you locate duplicates or conflicting entries. 

/etc/modprobe.d/blacklist-pcspkr.conf:blacklist pcspkr
Step 3 : Attempt to load the blacklisted module.
Command 
sudo modprobe pcspkr

This is a quick functional check. In real troubleshooting, you would also verify whether the module is built-in or already loaded. If it is already loaded, blacklisting does not remove it from the running kernel. 

modprobe: ERROR: Module pcspkr is blacklisted.
Step 4 : Create persistent options for loop.
Command 
echo 'options loop max_loop=16' | sudo tee /etc/modprobe.d/loop.conf

Module option files let you define a default parameter set that applies when the module is loaded. This is a common pattern for tuning driver behavior without editing bootloader config. 

options loop max_loop=16
Step 5 : Validate the option is recognized by the module loader.
Command 
modprobe -c | grep -E '^options\s+loop\b'

modprobe -c prints the resolved configuration after reading all relevant config files. This is the clearest way to confirm that your option line is being picked up. 

options loop max_loop=16
Alternate 
modinfo -p loop

modinfo -p lists parameters the module supports. It does not confirm your configured value, but it confirms the parameter exists and is valid for that module.

Step 6 : Rebuild initramfs so settings persist at early boot.
Command 
# Debian or Ubuntu
sudo update-initramfs -u

# RHEL or Fedora
sudo dracut -f

# Arch
sudo mkinitcpio -P

If a module is loaded from initramfs during early boot, your modprobe policy must be included there. Rebuilding initramfs makes sure your configuration and any relevant modules are packaged consistently for boot. 

update-initramfs: Generating /boot/initrd.img-$(uname -r)
Step 7 : Confirm pcspkr is not loaded and load loop.
Command 
lsmod | grep pcspkr

Blacklisting prevents loads, but it does not automatically remove a module that is already loaded. This check confirms the current runtime state.

Command 
sudo modprobe loop

Loading the module here confirms it is available and that module policy is not preventing normal module operations.

Command 
lsmod | grep -E '^loop(\s|$)'
loop                   45056  0

Reference

  • /etc/modprobe.d/*.conf : Persistent configuration files for module blacklists, aliases, and options.
  • blacklist <module> : Prevents the module from being loaded automatically and commonly blocks manual loads via modprobe.
  • options <module> param=value : Defines default parameters applied when the module is loaded.
  • modprobe -c : Prints resolved module loader configuration after reading all config files.
  • modinfo -p <module> : Lists supported module parameters.
  • lsmod : Lists loaded modules and usage counts.
  • Initramfs rebuild tools:
    • update-initramfs -u (Debian or Ubuntu)
    • dracut -f (RHEL or Fedora)
    • mkinitcpio -P (Arch)
