Lab 137: Time Sync & Logging Workflow

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Validate time sync on a RHEL host using timedatectl, systemctl, and chronyc, apply a safe chrony.conf change with a backup-first workflow, then confirm end-to-end logging by inspecting journald, verifying rsyslog, and proving delivery with logger into /var/log/messages.

rhel rhcsa services chrony journald rsyslog logging time

Scenario

A RHEL server is firing time drift alerts, and you need logs for a quick incident review. Your job is to confirm time/NTP status, validate that chronyd is healthy and syncing, apply a safe configuration change with a backup-first workflow, and then verify the logging pipeline by using journalctl and confirming rsyslog is receiving messages and writing to disk.

Operator context

In production, “service is running” is not the same as “service is working.” You need proof: time is synchronized to a source, logs exist where responders expect them, and a new message traverses the pipeline end-to-end.

Objective

  • Verify system time/NTP state with timedatectl.
  • Confirm chronyd is running and syncing to at least one source.
  • Back up /etc/chrony.conf, apply a safe makestep setting, and restart chrony.
  • Confirm chrony returns to an active state after the change.
  • Inspect journal storage usage and review recent chrony logs from journald.
  • Confirm rsyslog is running and writing messages to /var/log/messages.
  • Prove the pipeline by sending a test event with logger and verifying it lands on disk.

Walkthrough

Step 1: Check system time and NTP sync state.
Command 
timedatectl status

This is the fastest high-level signal: current time, time zone, and whether the system believes it is synchronized and running NTP. 

Local time: Sun 2026-01-25 07:12:19 EST
Universal time: Sun 2026-01-25 12:12:19 UTC
RTC time: Sun 2026-01-25 12:12:18
Time zone: America/New_York (EST, -0500)
System clock synchronized: yes
NTP service: active
RTC in local TZ: no
Step 2: Verify the chrony service is running.
Command 
sudo systemctl status chronyd --no-pager

Confirm the unit is active and note the process command line. If chrony is down, time can drift fast. 

● chronyd.service - NTP client/server
Loaded: loaded (/usr/lib/systemd/system/chronyd.service; enabled; preset: enabled)
Active: active (running) since Sun 2026-01-25 06:58:41 EST; 13min ago
Main PID: 1042 (chronyd)
Step 3: Confirm chrony is syncing to a source.
Command 
chronyc sources -v

A selected source (^*) is a strong signal the host is actively syncing rather than drifting. 

Number of sources = 2
MS Name/IP address         Stratum Poll Reach LastRx Last sample
===============================================================================
^* time1.example.net             2   6   377    32   -18us[ -41us] +/-  22ms
^+ time2.example.net             2   6   377    33   +12us[  -8us] +/-  24ms
Step 4: Back up the chrony configuration before making changes.
Command 
sudo cp -a /etc/chrony.conf /etc/chrony.conf.bak

This gives you an immediate rollback path if the change introduces an error or breaks sync.

Step 5: Append a safe makestep setting and verify it wrote.
Command 
echo 'makestep 1.0 3' | sudo tee -a /etc/chrony.conf

makestep allows chrony to step the clock when the offset is large (useful after boot or outage). The tee pattern avoids editor drift and gives you immediate feedback. 

makestep 1.0 3
Step 6: Restart chronyd to apply changes.
Command 
sudo systemctl restart chronyd

A restart is the cleanest way to ensure the new configuration is loaded.

Step 7: Confirm chronyd returned to an active state.
Command 
systemctl is-active chronyd
active

If this returns anything other than active, check systemctl status chronyd and fix the configuration before moving on.

Step 8: Check journald disk usage.
Command 
journalctl --disk-usage

This is a quick health check for journal storage pressure and helps you spot log storms or retention issues. 

Archived and active journals take up 72.0M in the file system.
Step 9: Review recent chronyd logs from the journal.
Command 
journalctl -u chronyd -n 8 --no-pager

This confirms the service restarted cleanly and selected a source after the configuration change. 

Jan 25 06:58:41 lab137 systemd[1]: Starting NTP client/server...
Jan 25 06:58:41 lab137 chronyd[1042]: chronyd version 4.4 starting (+CMDMON +NTP +REFCLOCK +RTC +PRIVDROP +SCFILTER +SIGND +ASYNCDNS +NTS)
Jan 25 06:58:43 lab137 chronyd[1042]: Selected source time1.example.net
Jan 25 07:11:58 lab137 systemd[1]: Stopping NTP client/server...
Jan 25 07:11:59 lab137 chronyd[1189]: Selected source time1.example.net
Step 10: Confirm rsyslog is running.
Command 
sudo systemctl status rsyslog --no-pager

On RHEL-style systems, rsyslog commonly writes to /var/log/messages. If rsyslog is down, responders may think “logs are missing” even though journald still has them. 

● rsyslog.service - System Logging Service
Loaded: loaded (/usr/lib/systemd/system/rsyslog.service; enabled; preset: enabled)
Active: active (running) since Sun 2026-01-25 06:58:38 EST; 13min ago
Main PID: 1019 (rsyslogd)
Step 11: Send a test syslog message.
Command 
logger -p user.notice 'test: logging pipeline OK'

This produces a new log event on demand so you can verify delivery end-to-end.

Step 12: Confirm the message landed in /var/log/messages.
Command 
sudo tail -n 5 /var/log/messages

You are looking for the test message. This proves rsyslog received the event and wrote it to disk. 

Jan 25 07:12:13 lab137 lab[pts/0]: test: logging pipeline OK

Reference

  • timedatectl status: view time zone, sync state, and NTP service state.
  • systemctl status chronyd, systemctl is-active chronyd: verify chrony service health.
  • chronyc sources -v: confirm chrony has reachable sources and a selected upstream.
  • /etc/chrony.conf: chrony configuration file (back up before editing).
  • journalctl --disk-usage: check journal storage consumption.
  • journalctl -u chronyd: view unit-specific logs from journald.
  • systemctl status rsyslog: verify syslog daemon status.
  • logger: generate a syslog event for pipeline testing.
  • /var/log/messages: common rsyslog destination on RHEL-like systems.
