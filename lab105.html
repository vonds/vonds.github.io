Lab 105: Installing, Configuring, and Managing a DHCP Server

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Install and configure a DHCP server to provide automatic IPv4 address assignment for a subnet, including lease ranges, DNS, and default gateway options. Validate service health, confirm interface binding, and inspect active lease records to verify end-to-end operation.

networking services core

Scenario

Your network requires automatic IP assignment for client devices. You are tasked with deploying a DHCP server, defining an IPv4 scope, setting DNS servers and a default gateway, and verifying that leases are being issued correctly.

Operator context

Treat DHCP changes as infrastructure changes. Validate syntax and service state before allowing clients to rely on the new scope, and always confirm the server is listening on the correct interface and subnet.

Objective

  • Install a DHCP server package on the system.
  • Configure an IPv4 subnet scope with a lease range, DNS servers, and default gateway.
  • Bind the DHCP service to the correct network interface.
  • Enable and start the service and confirm it is running.
  • Validate lease issuance by inspecting the leases database.

What You’ll Practice

  • Package installation workflows for network services across distros.
  • DHCP scope design and core options ( range , option domain-name-servers , option routers ).
  • Interface binding and service configuration differences across distributions.
  • Service lifecycle management with systemd.
  • Lease verification using the DHCP leases database.

Walkthrough

Step 1 : Install the DHCP server package.
Command 
# Debian/Ubuntu
sudo apt install isc-dhcp-server -y

# RHEL/Rocky/Alma
sudo dnf install dhcp-server -y

# Arch
sudo pacman -S dhcp

DHCP is typically provided by the ISC DHCP daemon ( dhcpd ) or a distro-packaged equivalent. Installing the package places the main configuration file on disk and registers the systemd service unit. 

# Confirm the server binary exists (one example):
command -v dhcpd
Step 2 : Edit the main DHCP configuration file.
Command 
sudo vim /etc/dhcp/dhcpd.conf
# or
sudo nano /etc/dhcp/dhcpd.conf

The DHCP configuration defines which subnets the server will manage and what options it will hand out to clients. At minimum you need a subnet declaration and a lease range. Most environments also include DNS and a default gateway ( routers ). 

subnet 192.168.50.0 netmask 255.255.255.0 {
  range 192.168.50.10 192.168.50.100;
  option domain-name-servers 8.8.8.8, 1.1.1.1;
  option routers 192.168.50.1;
  default-lease-time 600;
  max-lease-time 7200;
}
Step 3 : Bind the service to the correct network interface.
Command 
# Debian/Ubuntu (isc-dhcp-server)
echo 'INTERFACESv4="eth0"' | sudo tee /etc/default/isc-dhcp-server > /dev/null

# RHEL-style (dhcpd)
echo 'DHCPDARGS=eth0' | sudo tee /etc/sysconfig/dhcpd > /dev/null

DHCP should listen only on the interface connected to the client network. Binding prevents accidental lease offers on unintended networks and avoids confusing multi-interface hosts. The exact configuration path varies by distribution. 

# Confirm the interface exists and has the expected subnet:
ip -br addr show eth0
Step 4 : Enable and start the DHCP service.
Command 
# Debian/Ubuntu
sudo systemctl enable --now isc-dhcp-server

# RHEL-style
sudo systemctl enable --now dhcpd

Enabling the service ensures DHCP starts after boot. Starting it immediately lets you validate configuration correctness and listening behavior before any clients depend on it. 

# Confirm it is listening on UDP/67:
sudo ss -lunp | grep -E ':67\b'
Step 5 : Check service status and confirm the server is dispatching leases.
Command 
# Debian/Ubuntu
sudo systemctl status isc-dhcp-server

# RHEL-style
sudo systemctl status dhcpd

A healthy DHCP daemon will show an active running state and log that it is listening on the correct interface and subnet. If it fails to start, the most common causes are syntax errors, missing interface binding, or subnet mismatches with the server’s own IP configuration. 

# Typical success indicators:
# Listening on LPF/eth0/.../192.168.50.0/24
# Sending on   LPF/eth0/.../192.168.50.0/24
Step 6 : Inspect the leases file to verify clients are receiving addresses.
Command 
# Path varies by distro
sudo cat /var/lib/dhcp/dhcpd.leases
# or
sudo cat /var/lib/dhcpd/dhcpd.leases

The leases database is the authoritative record of which clients have been assigned addresses, including start/end times and client identifiers. Verifying leases is a direct proof that the server is functioning, and it also helps diagnose “client cannot get an IP” reports by confirming whether the request is reaching the server. 

lease 192.168.50.10 {
  starts 5 2025/12/19 22:05:01;
  ends 5 2025/12/19 22:15:01;
  binding state active;
  hardware ethernet 3c:52:82:19:aa:0f;
  client-hostname "laptop-qa";
}

Reference

  • dhcpd : DHCP server daemon that hands out IP leases and options to clients.
  • /etc/dhcp/dhcpd.conf : Main DHCP server configuration file (subnets, ranges, and options).
  • systemctl enable --now <service> : Enables a service at boot and starts it immediately.
  • ss -lunp : Shows UDP listening sockets and the owning process.
    • -l : Listening sockets only.
    • -u : UDP sockets.
    • -n : Do not resolve names.
    • -p : Show process info (requires privileges).
  • /var/lib/dhcp*/dhcpd.leases : Lease database used to confirm active and historical leases.
  • journalctl -u dhcpd : Service logs for DHCP startup, binding, and lease activity.
