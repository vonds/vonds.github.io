Lab 94: Configuring the Linux Firewall

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Configure host firewall policy to allow only required services while keeping a “deny by default” posture. Apply rules using either UFW (Debian/Ubuntu) or firewalld (RHEL/Fedora) and verify effective state.

security networking hardening

Scenario

You’re standardizing host firewall configuration. The system must allow SSH for administration and HTTP for a web service, deny other unsolicited inbound traffic, and support quickly blocking a hostile IP. You’ll implement the policy using UFW or firewalld based on distro, then verify rules and startup behavior.

Operator context

Firewall changes can lock you out. Confirm you have console access or an out-of-band path before enforcing restrictive inbound defaults on remote hosts.

Objective

  • Install the firewall tooling for your distro.
  • Enable the firewall and ensure it persists at boot.
  • Allow inbound SSH and HTTP explicitly.
  • Set a restrictive inbound default policy.
  • Reload/apply rules and inspect the active ruleset.
  • Block a specific IPv4 address.
  • Remove a previously-added rule and confirm state.

What You’ll Practice

  • UFW basics: enable, allow services, default policies, status.
  • firewalld basics: services, permanent rules, reload, inspection.
  • The difference between “configured” and “effective right now.”
  • Targeted blocks using UFW rules or firewalld rich rules.

Walkthrough

Step 1 : Install the firewall package.
Command 
# UFW (Debian/Ubuntu)
sudo apt install ufw -y
# OR (Arch)
sudo pacman -S ufw

# firewalld (RHEL/Fedora)
sudo dnf install firewalld -y

Install the tooling that matches the host OS family. Don’t try to run UFW commands on a host where you’re standardizing on firewalld (and vice versa).

Step 2 : Enable the firewall service.
Command 
# UFW
sudo ufw enable

# firewalld
sudo systemctl enable --now firewalld

This turns on policy enforcement. On remote systems, it’s safest to add SSH allow rules before enabling a restrictive inbound posture.

Step 3 : Allow inbound SSH.
Command 
# UFW
sudo ufw allow OpenSSH

# firewalld
sudo firewall-cmd --permanent --add-service=ssh

This is the “do not lock yourself out” rule. firewalld uses services (XML definitions) that map to ports/protocols.

Step 4 : Allow inbound HTTP.
Command 
# UFW
sudo ufw allow http

# firewalld
sudo firewall-cmd --permanent --add-service=http

This permits inbound port 80 traffic (service-based). If you’re serving HTTPS, you’d also add https.

Step 5 : Set a “deny inbound by default” posture.
Command 
# UFW
sudo ufw default deny incoming

# firewalld (example posture change)
sudo firewall-cmd --set-default-zone=drop

This establishes the baseline policy: only explicitly allowed inbound traffic gets in. Be deliberate when changing a firewalld default zone—zones affect how interfaces and sources are handled.

Step 6 : Reload/apply changes.
Command 
# UFW
sudo ufw reload

# firewalld
sudo firewall-cmd --reload

With firewalld, permanent changes require a reload to become active (unless you also add them to the runtime config).

Step 7 : View the current ruleset.
Command 
# UFW
sudo ufw status verbose

# firewalld
sudo firewall-cmd --list-all

This is your primary verification step: confirm SSH/HTTP are allowed and the default posture matches the intent.

Step 8 : Block a hostile IP address.
Command 
# UFW
sudo ufw deny from 203.0.113.10

# firewalld (rich rule example)
sudo firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=203.0.113.10 reject"

Use targeted blocks to stop abusive sources without changing the overall service policy. Rich rules provide expressive matching beyond simple service/port additions.

Step 9 : Remove a rule (example: HTTP allow).
Command 
# UFW
sudo ufw delete allow http

# firewalld
sudo firewall-cmd --permanent --remove-service=http

After removal, reload and re-check active state to ensure your policy matches the new requirement.

Step 10 : Confirm the firewall service is enabled at boot.
Command 
sudo systemctl is-enabled ufw
# OR
sudo systemctl is-enabled firewalld

This verifies persistence. “enabled” means the firewall will come up across reboots; “disabled” means policy may not be enforced after restart.

Reference

  • ufw : Simple host firewall management (front-end to nftables/iptables).
    • ufw enable : Enable firewall enforcement.
    • ufw allow <service> : Allow inbound service.
    • ufw default deny incoming : Deny inbound by default.
    • ufw status verbose : Inspect active policy.
  • firewalld : Zone-based firewall daemon (nftables backend on modern distros).
    • firewall-cmd --permanent --add-service=<name> : Add a service rule persistently.
    • firewall-cmd --reload : Apply permanent rules to runtime.
    • firewall-cmd --list-all : Inspect active zone configuration.
    • --add-rich-rule : Add more expressive match/action rules.
  • systemctl is-enabled <unit> : Confirm service enablement at boot.
