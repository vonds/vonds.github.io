Lab 103: Managing Roles with ansible-galaxy

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Use ansible-galaxy to discover, install, and reuse community roles, then generate a clean skeleton for your own role inside a project-local roles/ directory. Validate the role workflow end-to-end by running a role-based playbook against an inventory target.

automation roles ansible-galaxy

Scenario

You’re building modular infrastructure with Ansible and want a repeatable way to reuse (and eventually contribute) community roles. Your goal is to search Galaxy for a proven Nginx role, install it locally, consume it from a playbook, and then create a clean, production-ready skeleton for a custom internal role.

Operator context

Treat third-party roles like dependencies. You still validate what they do, pin versions when possible, and keep your project structure consistent so roles are discoverable and maintainable over time.

Objective

  • Create a project directory with a local roles/ path.
  • Search Ansible Galaxy for a community role and install one.
  • Use the installed role from a simple site playbook.
  • Generate a new custom role skeleton with ansible-galaxy init .

What You’ll Practice

  • Role discovery: ansible-galaxy search and reading role metadata at a glance.
  • Role installation: ansible-galaxy install and where roles land on disk.
  • Role consumption: a playbook using roles: with a Galaxy role name.
  • Role authoring basics: generating a clean role structure with ansible-galaxy init .

Walkthrough

Step 1 : Create a directory to hold your playbook and roles.
Command 
mkdir -p galaxy_lab/roles && cd galaxy_lab
  Directory 'galaxy_lab' created.
  roles directory created: galaxy_lab/roles

This is a simple “project layout” baseline. Your playbooks live at the project root, and your custom roles can live in ./roles (a common convention).

Step 2 : Search for a community role to install.
Command 
ansible-galaxy search nginx
  Found 25 roles matching 'nginx'

  Name                                 Description
  ----                                 -----------
  geerlingguy.nginx                    Nginx for Linux.
  nginxinc.nginx                       Official NGINX role (install/config).
  idealista.nginx_role                 Installs and configures Nginx.
  jdauphant.nginx                      Install and configures nginx
  ...

Search is your fast discovery step. In real work, you’d also review README, supported platforms, defaults, and whether the role is actively maintained.

Step 3 : Install the role using ansible-galaxy.
Command 
ansible-galaxy install geerlingguy.nginx
  Starting galaxy role install process
  - downloading role 'nginx', owned by geerlingguy
  - downloading role from https://github.com/geerlingguy/ansible-role-nginx/archive/refs/heads/master.tar.gz
  - extracting geerlingguy.nginx to /home/lab/.ansible/roles/geerlingguy.nginx
  - geerlingguy.nginx (master) was installed successfully

By default, Galaxy installs into your user role path (often ~/.ansible/roles ). In production, you typically pin versions and may install dependencies into a project-local roles path for repeatability.

Step 4 : Create a playbook to use the downloaded role.
Command 
nano site.yml
# OR
vim site.yml
File 
- hosts: all
  become: true
  roles:
    - geerlingguy.nginx

This is the simplest role consumption pattern: list the role name under roles: and Ansible runs its default entry point (tasks/main.yml).

Step 5 : Run the playbook that uses the role.
Command 
ansible-playbook -i inventory.ini site.yml
  PLAY [all] *********************************************************************

  TASK [Gathering Facts] *********************************************************
  ok: [target1]

  TASK [geerlingguy.nginx : Include OS-specific variables] ************************
  ok: [target1]

  TASK [geerlingguy.nginx : Ensure nginx is installed] ****************************
  changed: [target1]

  TASK [geerlingguy.nginx : Copy nginx configuration in place.] *******************
  changed: [target1]

  TASK [geerlingguy.nginx : Ensure nginx is started and enabled at boot.] *********
  changed: [target1]

  PLAY RECAP *********************************************************************
  target1                    : ok=6    changed=3    unreachable=0    failed=0    skipped=0    rescued=0    ignored=0

This confirms the role installs, applies configuration, and manages the service. If you need a fast validation without changes, you can run --check (dry run) or --syntax-check for quick parsing validation.

Step 6 : Create a custom role skeleton using init.
Command 
ansible-galaxy init roles/myrole
  - Role roles/myrole was created successfully
  - Created roles/myrole/tasks/main.yml
  - Created roles/myrole/handlers/main.yml
  - Created roles/myrole/defaults/main.yml
  - Created roles/myrole/vars/main.yml
  - Created roles/myrole/meta/main.yml
  - Created roles/myrole/README.md
  - Created roles/myrole/files/
  - Created roles/myrole/templates/
  - Created roles/myrole/tests/inventory
  - Created roles/myrole/tests/test.yml

This is your starting point for internal roles: predictable structure, clear defaults, and a place for templates/files/tests. From here, you move logic into tasks/ and only expose necessary knobs through defaults/ .

Reference

  • ansible-galaxy search <term> : Search Ansible Galaxy for roles matching a keyword.
  • ansible-galaxy install <namespace.role> : Download and install a role into the default roles path.
  • ansible-galaxy init <path> : Generate a standard role skeleton with recommended directories.
  • roles: : Playbook section used to include and execute roles.
