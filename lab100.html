Lab 100: Agent-Based Configuration with ansible-pull

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Configure a host to pull and apply its desired state from a Git repository using ansible-pull. You’ll run a local self-apply, then schedule it so the node continuously converges without a central controller.

ansible ansible-pull git cron desired-state

Scenario

You want Linux systems to self-configure by pulling their desired state from a Git repository and applying it locally. This is useful when hosts can reach Git but a central controller can’t reliably reach them (branch networks, NAT, limited inbound access, or “pull-only” environments).

Operator context

With ansible-pull, the node is the “controller.” It clones (or updates) a repo, runs a playbook against localhost, and repeats on a schedule so it stays converged.

Objective

  • Install Ansible and Git on the managed node.
  • Inspect a repo-hosted playbook intended for localhost configuration.
  • Run ansible-pull to apply configuration locally.
  • Schedule ansible-pull to run every 30 minutes.
  • Prepare a log path to capture scheduled run output.

What You’ll Practice

  • Agent-based config model with ansible-pull
  • Git-backed desired state
  • Local execution targeting localhost
  • Recurring convergence via cron
  • Operational logging for scheduled automation

Walkthrough

Step 1: Install Ansible and Git.
Command 
sudo dnf install ansible git -y
# (or apt/pacman equivalents)

Both are required: Git pulls your desired state, Ansible applies it.

Step 2: Clone the remote repository.
Command 
git clone https://github.com/example/ansible-lab100.git

In production you’d typically point at an internal repo or a locked-down private repo.

Step 3: Inspect the playbook that targets localhost.
Command 
cat ansible-lab100/site.yml
# or
less ansible-lab100/site.yml

For ansible-pull, the pattern is usually hosts: localhost with become: true, since the node configures itself.

Step 4: Apply the desired state locally using ansible-pull.
Command 
sudo ansible-pull -U https://github.com/example/ansible-lab100.git -i localhost site.yml

This clones (or updates) the repo into the default pull directory and runs the playbook against localhost. On later runs, you should see fewer changes as the system converges.

Step 5: Schedule the node to self-apply every 30 minutes.
Command 
crontab -e
Cron entry 
*/30 * * * * /usr/bin/ansible-pull -U https://github.com/example/ansible-lab100.git -i localhost site.yml

This is the simplest convergence loop. In production, you’ll usually add logging (next step) and sometimes a lock to prevent overlap if a run takes longer than the interval.

Step 6: Prepare a log file for scheduled run output.
Command 
sudo mkdir -p /var/log/ansible
sudo touch /var/log/ansible/pull.log

At minimum, capture output so you can verify runs and troubleshoot failures quickly.

Ops Notes

Make cron safer (typical production pattern)

Cron often runs with a minimal environment. Use absolute paths, and redirect stdout/stderr to a log. Also consider a lock to prevent overlapping runs.

Example cron line (safer) 
*/30 * * * * flock -n /var/lock/ansible-pull.lock /usr/bin/ansible-pull -U https://github.com/example/ansible-lab100.git -i localhost site.yml >> /var/log/ansible/pull.log 2>&1
  • If you’re pulling from a private repo, the usual approaches are deploy keys, a machine identity (SSO), or a secret manager flow. Keep credentials out of the repo.
  • For larger setups, pairing ansible-pull with systemd timers can be cleaner than cron.
