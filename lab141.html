Lab 141: Postfix Queue & Time/Journal Workflow

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Forward root mail to two addresses by updating /etc/aliases and rebuilding the aliases database. Validate Postfix health, inspect the mail queue, and use postcat to view a queued message by ID. Finish with quick time and journald checks to confirm the system signals you would capture during incident review.

rhel rhcsa postfix mail aliases chrony journald

Scenario

Ops wants all root mail forwarded to two addresses. There is also a queued message that needs inspection to confirm what is being sent and why it is stuck.

Why this matters

Root mail often contains cron output, service alerts, and local security notices. If it is not forwarded correctly, incidents are missed. Queue inspection is a core skill for diagnosing outbound mail failures.

Objective

  • Edit /etc/aliases to forward root to two recipients.
  • Rebuild the aliases database with newaliases.
  • Confirm Postfix is active.
  • Inspect the queue with mailq or postqueue -p.
  • View a queued message with postcat -q.
  • Confirm time zone state using timedatectl.
  • Confirm chrony config file path exists.
  • Pull kernel messages from journald with journalctl -k.
  • Verify the alias line was applied.

Walkthrough

Step 1: Update the aliases file.
Command 
sudo vim /etc/aliases

Add or update the root alias so messages are forwarded to both recipients. 

root: admin@example.com, webmaster@example.com
Note

The aliases database is not updated automatically. Editing the file alone does not apply the change.

Step 2: Rebuild the aliases database.
Command 
sudo newaliases

This compiles /etc/aliases into the database file that Postfix actually uses. 

/etc/aliases: 86 aliases, longest 52 bytes, 948 bytes total
Step 3: Confirm Postfix is active.
Command 
systemctl is-active postfix

You want a clean “active” signal before spending time on queue analysis. 

active
Step 4: Inspect the mail queue.
Command 
mailq

Identify stuck messages, note the queue ID, and capture sender and recipient to guide the next step. 

-Queue ID-  --Size-- ----Arrival Time---- -Sender/Recipient-------
A1B2C3D4E5*     1468 Sun Jan 25 07:10:41  root@lab141
                                         admin@example.com
-- 2 Kbytes in 1 Request.
Step 5: View the queued message contents.
Command 
sudo postcat -q A1B2C3D4E5

postcat -q lets you inspect what is in the queue without attempting delivery. This is useful for confirming message headers, recipients, and the subject line during triage. 

*** ENVELOPE RECORDS active ***
message_size:           1468
message_arrival_time:   Sun Jan 25 07:10:41 2026
sender:                 root@lab141
recipient:              admin@example.com
*** MESSAGE CONTENTS active ***
Subject: RHCSA-LAB141 queued test

This is a queued test message.
Step 6: List available time zones.
Command 
timedatectl list-timezones | head -n 5

This confirms the system has the time zone database available and shows how to safely limit output. 

Africa/Abidjan
Africa/Accra
Africa/Addis_Ababa
Africa/Algiers
Africa/Asmara
Step 7: Confirm current time and time zone.
Command 
timedatectl

Capture the exact time zone and sync state as part of incident context, especially when mail timestamps and logs need to correlate with external systems. 

Time zone: America/New_York (EST, -0500)
System clock synchronized: yes
NTP service: active
Step 8: Confirm the chrony configuration file path.
Command 
ls -l /etc/chrony.conf

In real incidents, you often need to confirm where the active config lives before making changes or collecting evidence. 

-rw-r--r--. 1 root root 1247 Jan 25 06:58 /etc/chrony.conf
Step 9: Pull kernel messages from the journal.
Command 
journalctl -k -n 5

Kernel messages can provide useful context such as link state, driver resets, and audit signals that explain service behavior. 

Jan 25 06:58:31 lab141 kernel: e1000e 0000:00:03.0 eth0: Link is Up 1000 Mbps Full Duplex
Jan 25 07:12:03 lab141 kernel: audit: type=1100 audit(...): pid=1 uid=0 ... unit=postfix ...
Step 10: Verify the root alias line is present.
Command 
grep '^root:' /etc/aliases

This is a quick verification that the correct alias line exists after edits. 

root: admin@example.com, webmaster@example.com

Common breakpoints

Alias change does not take effect

You edited /etc/aliases but did not run newaliases. Rebuild the database and try again.

mailq shows “Mail queue is empty”

That is not an error. It just means there is nothing queued on this host at the moment.

postcat fails: “No such file or directory”

The queue ID may be wrong, or the message was delivered/removed. Re-run mailq and copy the exact ID.

Time looks correct but incidents still show drift

Check the NTP sync state, and confirm chrony sources with chronyc sources -v if needed.

Cleanup checklist

Leave the system in a safe state. If you modified aliases for lab testing, revert the line to the prior value.

Commands 
# If you changed /etc/aliases for the lab, revert as needed
sudo vim /etc/aliases
sudo newaliases

# Optional: verify Postfix state
systemctl is-active postfix || systemctl status postfix --no-pager

Reference

  • /etc/aliases: local alias map used for address rewriting.
  • newaliases: rebuild alias database for Postfix to use.
  • systemctl is-active postfix: quick service health signal.
  • mailq, postqueue -p: inspect the Postfix queue.
  • postcat -q QUEUEID: view queued message content without delivery attempts.
  • timedatectl: time zone and NTP sync state.
  • /etc/chrony.conf: chrony configuration file path.
  • journalctl -k: kernel messages from the system journal.
