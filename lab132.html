Lab 132: Fix Slow Hostname Lookups (Reverse DNS)

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Diagnose slow SSH and sudo caused by hostname resolution delays, then restore fast lookups by adjusting the host resolution order. Prove the fix with before-and-after timing evidence.

networking troubleshooting dns

Scenario

Users complain that “SSH and sudo feel slow on this server.” You suspect hostname resolution delays due to reverse DNS or resolver timeouts. Your job is to confirm the delay, identify the root cause, and fix host resolution so admin workflows are responsive again.

Operator context

This is an ops reliability issue, not a performance mystery. The goal is to collect timing evidence, remove the timeout path from the hot loop, and prove the improvement with the same verification commands.

Objective

  • Measure a slow hostname lookup using time and getent.
  • Identify which DNS server the host is using.
  • Prove DNS is timing out by querying the server directly.
  • Inspect host lookup order in /etc/nsswitch.conf.
  • Prefer files before dns for host lookups.
  • Re-test lookup speed and verify the issue is resolved.

What You’ll Practice

  • Measuring lookup latency with time and getent.
  • Inspecting resolver configuration via resolvectl or /etc/resolv.conf.
  • Testing DNS server reachability using dig with an explicit server.
  • Auditing the hosts: lookup chain in /etc/nsswitch.conf.
  • Making a safe, targeted config edit and verifying impact with repeatable checks.

Walkthrough

Step 1 : Measure how long a hostname lookup takes.
Command 
time getent hosts portal.lab.local

getent follows the system’s configured name service order. Timing this call gives you hard evidence that name resolution is the slow path. 

10.0.2.50   portal.lab.local

real    0m2.01s
user    0m0.00s
sys     0m0.00s
Step 2 : Check which DNS server(s) this host is using.
Command 
resolvectl status

This tells you where the system is sending DNS queries. A dead or unreachable resolver will cause retries and timeouts that look like “random slowness.” 

Link 2 (eth0)
     DNS Servers: 10.0.2.99
Current DNS Server: 10.0.2.99
Step 3 : Prove the configured DNS server is timing out.
Command 
dig @10.0.2.99 portal.lab.local

Querying the server explicitly isolates the failure to the DNS server itself, not your application, not SSH, and not the interface. 

;; communications error to 10.0.2.99#53: timed out
;; communications error to 10.0.2.99#53: timed out
;; no servers could be reached
Step 4 : Inspect the host lookup order.
Command 
grep -n '^hosts:' /etc/nsswitch.conf

This line defines the order for host resolution. If DNS is first and your resolver is timing out, every lookup pays the timeout penalty before falling back. 

12:hosts: dns files myhostname
Step 5 : Change the lookup order to prefer files before dns.
Command 
sudo vim /etc/nsswitch.conf

Edit the hosts: line to: hosts: files dns myhostname. This avoids DNS timeouts for names that can be satisfied locally (or for cases where local entries are sufficient for admin workflows). 

(editor opened)
(updated hosts: line and saved)
Step 6 : Re-measure lookup time and confirm the improvement.
Command 
time getent hosts portal.lab.local

You should see the lookup return quickly now. This is the before-and-after evidence that the slow path was DNS-first resolution combined with a timing-out resolver. 

10.0.2.50   portal.lab.local

real    0m0.01s
user    0m0.00s
sys     0m0.00s
Step 7 : Quick sanity check: resolve the host and validate service responsiveness.
Command 
curl -I http://portal.lab.local

This confirms resolution and a basic application path in one quick check. In practice, it’s the kind of “am I done yet” sanity test you run before handing a system back to users. 

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Server: nginx

Reference

  • getent hosts NAME : Resolves a name using the system’s NSS configuration.
  • time COMMAND : Measures elapsed time so you can prove latency and improvement.
  • resolvectl status : Displays active resolver configuration under systemd-resolved.
  • dig @SERVER NAME : Queries a specific DNS server to test reachability and response.
  • /etc/nsswitch.conf : Controls lookup order for services such as hosts.
  • curl -I URL : Fetches HTTP headers to confirm connectivity and basic service response.
