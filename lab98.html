Lab 98: Installing, Configuring, and Managing Ansible

« »
Lab Index
  • Scenario
  • Objective
  • What You'll Practice
  • Walkthrough
  • Reference

Install Ansible, build a simple inventory, validate connectivity with an ad-hoc ping, and run a YAML playbook to configure web hosts. This lab focuses on practical day-one operations: install, verify, target hosts, apply a change, and confirm idempotency.

ansible automation inventory playbooks

Scenario

You need to deploy and configure Ansible to automate routine server tasks. Your starting point is a small fleet: one web host and one database host, both reachable on a private network.

Operator context

The goal here is repeatability. You’re building a minimal but realistic Ansible workflow: define targets, validate connectivity, apply a change via playbook, and confirm re-runs don’t cause unnecessary changes.

Objective

  • Install Ansible and verify the runtime and config location.
  • Create a simple inventory with web and db host groups.
  • Run an ad-hoc ping module check across all hosts.
  • Write a YAML playbook to install and manage nginx on web hosts.
  • Gather facts using the setup module.
  • Clean up using an ad-hoc package module removal.

What You’ll Practice

  • Installing and validating tooling: ansible --version
  • Inventory management: ~/inventory, groups ([web], [db])
  • Ad-hoc operations: ansible all -m ping, ansible all -m setup
  • Playbooks and idempotency: ansible-playbook, package, service

Walkthrough

Step 1: Install Ansible.
Command 
sudo dnf install ansible -y
# Debian/Ubuntu:
# sudo apt install ansible -y
# Arch:
# sudo pacman -S ansible

On RHEL-like systems you’ll usually see ansible-core and the full ansible package pulled in through the distro repositories.

Step 2: Verify installation and config path.
Command 
ansible --version

Confirm the reported config file path and the python runtime. This is also your quick sanity check when behavior differs between environments.

Step 3: Create a basic inventory.
Command 
nano ~/inventory
# or
vim ~/inventory
~/inventory (example) 
[web]
192.168.122.10

[db]
192.168.122.11

This is intentionally minimal: groups first, then one host per line. In a real environment you’d also add host vars (user, key, port) or set defaults in ansible.cfg.

Step 4: Validate connectivity with an ad-hoc ping.
Command 
ansible all -i ~/inventory -m ping

This checks “can Ansible reach the hosts and run a module,” which is the baseline before running any playbook.

Step 5: Create a playbook to install nginx on web hosts.
Command 
nano install_nginx.yml
# or
vim install_nginx.yml
install_nginx.yml (initial) 
- name: Install nginx
  hosts: web
  become: yes
  tasks:
    - name: Install nginx
      package:
        name: nginx
        state: present

This is the cleanest “first playbook” because the package module abstracts the underlying package manager.

Step 6: Run the playbook.
Command 
ansible-playbook -i ~/inventory install_nginx.yml

The first run should show changed when nginx gets installed. That’s expected: you applied a new state.

Step 7: Ensure nginx is enabled and running.
Command 
nano install_nginx.yml
# or
vim install_nginx.yml
install_nginx.yml (updated) 
- name: Install nginx
  hosts: web
  become: yes
  tasks:
    - name: Install nginx
      package:
        name: nginx
        state: present

    - name: Ensure nginx is running
      service:
        name: nginx
        state: started
        enabled: yes

This is where playbooks become “ops-friendly”: install the thing, ensure it’s running, ensure it persists across reboot.

Step 8: Re-run to confirm idempotency.
Command 
ansible-playbook -i ~/inventory install_nginx.yml

A clean re-run should show mostly ok. You only want changed when something actually needed correction.

Step 9: Gather facts from hosts.
Command 
ansible all -i ~/inventory -m setup

This returns a large facts payload. In real playbooks, you’d reference facts to branch logic (OS family, IP addresses, disk layout, etc.).

Step 10: Clean up (remove nginx from web hosts).
Command 
ansible web -i ~/inventory -m package -a "name=nginx state=absent" -b

This mirrors your script lab: apply a change, verify it happened, and reset the environment for the next repetition.

Reference

  • Install + verify
    • sudo dnf install ansible -y
    • ansible --version
  • Inventory + ad-hoc
    • ansible all -i ~/inventory -m ping
    • ansible all -i ~/inventory -m setup
  • Playbooks
    • ansible-playbook -i ~/inventory install_nginx.yml
    • package: install/remove packages
    • service: start/enable services
  • Cleanup
    • ansible web -i ~/inventory -m package -a "name=nginx state=absent" -b
