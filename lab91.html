Lab 91: OpenLDAP Server Installation and Initial Configuration

« »
Lab Index

  • Objective

  • Reference

Install and bring up an OpenLDAP directory service, establish a base DN, and create the initial directory structure for basic identity management. Validate service health and query the directory to confirm the server is responding with expected entries.

identity directory services

Scenario

Install and configure an OpenLDAP directory server for basic user management. You will install packages, generate a password hash, configure a base DN, and test access using LDAP client tools.

Operator context

LDAP becomes foundational quickly. Treat the base DN, schema state, and admin bind as critical baseline configuration and verify each step before you start adding users and groups.

Objective

  • Install OpenLDAP server components and client utilities.
  • Configure the directory domain and base DN and set the admin password.
  • Generate a password hash for use in configuration.
  • Validate the slapd service is running under systemd.
  • Load core schema (when required by the distro layout).
  • Create and load a base LDIF to initialize directory structure.
  • Query the directory to confirm entries are present.

What You’ll Practice

  • Package installation for directory services and LDAP client tools.
  • Initial slapd configuration concepts: domain, base DN, admin DN.
  • Generating SSHA password hashes with slappasswd .
  • Schema loading via LDIF with SASL EXTERNAL over ldapi:/// .
  • Authoring and applying LDIF files with ldapadd .
  • Read-only verification with ldapsearch .

Walkthrough

Step 1 : Install OpenLDAP server and client utilities.
Command 
# Debian/Ubuntu
sudo apt install slapd ldap-utils -y

# RHEL/Fedora family
sudo dnf install openldap-servers openldap-clients -y

# Arch
sudo pacman -S openldap

You need both the directory service and the client utilities so you can configure and validate the directory from the CLI.

Step 2 : Reconfigure slapd and define domain/base DN (Debian/Ubuntu flow).
Command 
sudo dpkg-reconfigure slapd

On Debian/Ubuntu, dpkg-reconfigure drives the initial slapd configuration prompts (domain, organization, admin password, database backend). On other distributions the initialization steps differ, but the goal is the same: a defined base DN and an admin bind DN you can use to manage the directory. 

# Outcome (example):
slapd configured with your specified domain and base DN.
Step 3 : Generate an LDAP password hash.
Command 
slappasswd

slappasswd produces an SSHA hash suitable for LDAP password fields. This is commonly used when you are writing LDIF and need a hashed value rather than a plaintext secret. 

{SSHA}REDACTED_HASH_OUTPUT
Step 4 : Verify slapd is running.
Command 
sudo systemctl status slapd

This is your baseline service-health check. If slapd is not active, schema and LDIF operations are premature. 

# Expected pattern:
slapd is active (running)
Step 5 : Load core schema if required.
Command 
# Debian/Ubuntu schema path example:
ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/ldap/schema/core.ldif

# RHEL/Fedora schema path example:
ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/core.ldif

Schema defines object types and attributes your directory can store. Some setups have core schema loaded by default; others require you to load it explicitly using SASL EXTERNAL over the local LDAPI socket. 

core.ldif schema loaded successfully.
Step 6 : Create a base LDIF for the domain.
Command 
nano ~/base.ldif
# or
vim ~/base.ldif

This LDIF initializes the base DN entry for your directory. Your actual DN should match the domain you configured (the lab uses dc=example,dc=com as a placeholder). 

dn: dc=example,dc=com
objectClass: top
objectClass: dcObject
objectClass: organization
o: Example Company
dc: example
Step 7 : Add the base DN using ldapadd.
Command 
ldapadd -x -D cn=admin,dc=example,dc=com -W -f ~/base.ldif

This performs a simple bind ( -x ) as the admin DN and adds the LDIF content into the directory. In production you typically use TLS and may bind to a dedicated directory manager DN depending on the deployment model. 

Base DN structure added to LDAP directory.
Step 8 : Verify directory content with ldapsearch.
Command 
ldapsearch -x -LLL -b dc=example,dc=com

This performs a simple bindless search (read-only) and returns entries under the base DN. It’s a quick validation that slapd is responding and your base entries exist. 

# Expected pattern:
dn: dc=example,dc=com
objectClass: dcObject
objectClass: organization
...

Reference

  • slapd : OpenLDAP directory server daemon.
  • ldap-utils / openldap-clients : LDAP client tools (includes ldapadd/ldapsearch).
  • dpkg-reconfigure slapd : Debian/Ubuntu interactive initialization for slapd.
  • slappasswd : Generates password hashes for LDAP configuration/LDIF use.
  • ldapadd -Y EXTERNAL -H ldapi:/// -f <file.ldif> : Loads schema/LDIF using SASL EXTERNAL over the local LDAP socket.
  • ldapadd -x -D <binddn> -W -f <file.ldif> : Adds entries using a simple bind as a specified DN.
  • ldapsearch -x -LLL -b <baseDN> : Queries entries under a base DN with clean output format.
