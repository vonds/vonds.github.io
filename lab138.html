Lab 138: Printing & Mail Queue Workflow

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Validate printing and outbound mail on a RHEL host by verifying CUPS, submitting a test job, confirming queue state, then triaging the Postfix mail queue, forcing delivery attempts, deleting a stuck message, and confirming evidence in journald with unit logs.

rhel rhcsa services cups printing postfix mail journald triage

Scenario

A helpdesk ticket reports printing is “working sometimes,” and outbound mail is backing up. You need to validate the print service, submit a test job, confirm the job appears in the queue, then triage Postfix by inspecting the mail queue, forcing a delivery attempt, deleting a stuck message, and capturing evidence from the journal.

Operator context

Your goal is fast proof. Confirm services are healthy, create a known-good test event (print job, queue flush), then verify the system recorded your actions in unit logs.

Objective

  • Verify cups is running and enabled.
  • Confirm printers exist and identify the default destination.
  • Submit a test print job and confirm the job appears in the print queue.
  • Verify Postfix is active.
  • Inspect the mail queue and identify deferred/stuck messages.
  • Flush the mail queue to force a delivery attempt.
  • Delete a specific stuck message by queue ID using postsuper.
  • Confirm the queue state after remediation.
  • Capture evidence with journalctl -u postfix.

Walkthrough

Step 1: Verify the CUPS service is running.
Command 
sudo systemctl status cups --no-pager

Confirm cups is active. If it is down, printing will be intermittent or fail entirely. 

● cups.service - CUPS Scheduler
Loaded: loaded (/usr/lib/systemd/system/cups.service; enabled; preset: enabled)
Active: active (running) since Sun 2026-01-25 06:41:12 EST; 31min ago
Main PID: 1126 (cupsd)
Step 2: List printers and identify the default destination.
Command 
lpstat -p -d

You want to see at least one printer and a default destination. This determines where test jobs go. 

printer Office_Printer is idle. enabled since Sun 25 Jan 2026 06:41:14 AM EST
system default destination: Office_Printer
Step 3: Submit a test job to the default printer.
Command 
lp /etc/hosts

Creating a fresh job gives you a known-good signal to watch through the queue. 

request id is Office_Printer-23 (1 file(s))
Step 4: Confirm the print queue shows the job.
Command 
lpq

You are looking for the job ID and state. If jobs pile up or remain active too long, investigate printer connectivity and backend filters next. 

Office_Printer is ready
Rank   Owner   Job  File(s)                         Total Size
active lab     23   hosts                           1024 bytes
Step 5: Verify Postfix is running.
Command 
systemctl is-active postfix

If Postfix is inactive, mail will not move and the queue will grow quickly. 

active
Step 6: Inspect the Postfix mail queue.
Command 
mailq

Identify deferred entries and read the reason string. In this scenario, one message is stuck due to a connection timeout to the upstream MX on port 25. 

-Queue ID-  --Size-- ----Arrival Time---- -Sender/Recipient-------
3F2C61A012*     1468 Sun Jan 25 07:06:11  alerts@example.com
                                         ops@example.com
                                         (connect to mx.example.com[203.0.113.25]:25: Connection timed out)
1F9B77C0A9       912 Sun Jan 25 07:07:04  root@lab138
                                         admin@example.com
-- 2 Kbytes in 2 Requests.
Step 7: Force a delivery attempt.
Command 
postqueue -f

Flushing the queue is a fast way to validate whether the failure persists right now.

Step 8: Delete the stuck message by queue ID.
Command 
sudo postsuper -d 3F2C61A012

This removes a specific entry. In real incidents, document why you deleted it and capture the failure reason first. 

postsuper: Deleted: 1 message
Step 9: Re-check the queue to confirm the deletion.
Command 
postqueue -p

Confirm the stuck message is gone and the queue is smaller. 

-Queue ID-  --Size-- ----Arrival Time---- -Sender/Recipient-------
1F9B77C0A9       912 Sun Jan 25 07:07:04  root@lab138
                                         admin@example.com
-- 1 Kbytes in 1 Request.
Step 10: Capture evidence in the journal for Postfix activity.
Command 
journalctl -u postfix -n 8 --no-pager

Unit logs provide the incident timeline: enqueue, delivery attempts, deferral reason, flush, and deletion. 

Jan 25 07:06:11 lab138 postfix/qmgr[1288]: 3F2C61A012: from=, size=1468, nrcpt=1 (queue active)
Jan 25 07:06:44 lab138 postfix/smtp[1410]: 3F2C61A012: to=, relay=mx.example.com[203.0.113.25]:25, delay=33, dsn=4.4.1, status=deferred (connect to mx.example.com[203.0.113.25]:25: Connection timed out)
Jan 25 07:12:20 lab138 postfix/postqueue[1467]: warning: flush request received
Jan 25 07:12:30 lab138 postfix/postsuper[1481]: Deleted: 3F2C61A012
Jan 25 07:12:31 lab138 postfix/qmgr[1288]: 3F2C61A012: removed

Reference

  • systemctl status cups: verify the print scheduler is active.
  • lpstat -p -d: list printers and show the default destination.
  • lp /etc/hosts, lpr /etc/hosts: submit a test print job.
  • lpq, lpstat -o: inspect print job queue state.
  • systemctl is-active postfix: confirm mail service is running.
  • mailq, postqueue -p: view the Postfix queue and deferral reasons.
  • postqueue -f: flush the queue (force delivery attempt).
  • postsuper -d QUEUEID: delete a specific message from the queue.
  • journalctl -u postfix: view Postfix unit logs for timeline/evidence.
