Lab 134: Fix DNS Failure Caused by Bad resolv.conf

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Confirm the host can reach the internet by IP, then isolate the outage to DNS. Fix the resolver configuration persistently via NetworkManager and verify name resolution is restored.

networking troubleshooting dns

Scenario

Users report: “The server can ping 1.1.1.1, but anything using hostnames fails.” You suspect a resolver misconfiguration on this host and need to restore reliable DNS quickly and persistently.

Operator context

Do not “fix” DNS by hand-editing /etc/resolv.conf on a NetworkManager system. Treat it as generated output and correct the upstream config so the fix survives reboots and DHCP renewals.

Objective

  • Confirm a default route exists.
  • Prove raw internet connectivity works by IP.
  • Reproduce the DNS failure using getent.
  • Inspect current resolver configuration in /etc/resolv.conf.
  • Set working DNS servers persistently with nmcli.
  • Apply the change and confirm /etc/resolv.conf updates.
  • Verify name resolution works again.

What You’ll Practice

  • Separating routing/path issues from DNS issues.
  • Using ip route and ping as fast health checks.
  • Validating name resolution through NSS with getent.
  • Reading and interpreting /etc/resolv.conf.
  • Applying persistent DNS config via NetworkManager using nmcli.

Walkthrough

Step 1 : Confirm the host has a default route.
Command 
ip route
# OR
ip r

DNS will never work reliably without a valid path off the subnet. Confirm the default route before you chase resolver settings. 

default via 192.168.50.1 dev eth0 proto dhcp metric 100
192.168.50.0/24 dev eth0 proto kernel scope link src 192.168.50.20 metric 100
Step 2 : Prove internet connectivity works by IP (bypass DNS).
Command 
ping -c 2 1.1.1.1
# OR
ping -c 2 8.8.8.8

This isolates the issue away from routing. If the ping works, your next suspect is name resolution. 

PING 1.1.1.1 (1.1.1.1) 56(84) bytes of data.
64 bytes from 1.1.1.1: icmp_seq=1 ttl=56 time=16.2 ms
64 bytes from 1.1.1.1: icmp_seq=2 ttl=56 time=15.9 ms

--- ping statistics ---
2 packets transmitted, 2 received, 0% packet loss
Step 3 : Demonstrate DNS failure using NSS (getent).
Command 
getent hosts example.com
# OR
getent ahostsv4 example.com

getent follows the system’s configured name service switch order and gives you an operator-friendly signal: resolution either works or it does not. 

(no output)
getent: Name or service not known
Step 4 : Inspect the current resolver configuration.
Command 
cat /etc/resolv.conf
# OR
sudo cat /etc/resolv.conf

This host is pointing at 127.0.0.1 as its DNS server, which only works if a local resolver is actually running and configured. In this incident, it is not. 

# Generated by NetworkManager
search lab.local
nameserver 127.0.0.1
Step 5 : Set working DNS servers persistently using NetworkManager.
Command 
sudo nmcli con mod "System eth0" ipv4.dns "1.1.1.1 8.8.8.8"

This changes the source of truth so the fix survives link resets, DHCP renewals, and reboots. You are not editing /etc/resolv.conf directly.

Step 6 : Bounce the connection to apply the change.
Command 
sudo nmcli con down "System eth0" && sudo nmcli con up "System eth0"

Bringing the connection down and back up causes NetworkManager to regenerate resolver state using the updated DNS servers. 

Connection 'System eth0' successfully deactivated (D-Bus active path: /org/freedesktop/NetworkManager/ActiveConnection/7)
Connection 'System eth0' successfully activated (D-Bus active path: /org/freedesktop/NetworkManager/ActiveConnection/8)
Step 7 : Confirm /etc/resolv.conf contains real nameservers.
Command 
cat /etc/resolv.conf
# OR
sudo cat /etc/resolv.conf

This is your evidence that the persistent config is now generating correct resolver output. 

# Generated by NetworkManager
search lab.local
nameserver 1.1.1.1
nameserver 8.8.8.8
Step 8 : Verify name resolution works again.
Command 
getent hosts example.com
# OR
getent ahostsv4 example.com

Resolution is the primary success signal. If this works, tools like dnf, curl, and package repositories should behave normally again. 

93.184.216.34   example.com

Reference

  • ip route : Shows the kernel routing table, including the default route.
  • ping -c N IP : Confirms path connectivity without involving DNS.
  • getent hosts NAME : Resolves names through NSS and reports system-level resolution.
  • /etc/resolv.conf : Resolver configuration used by libc; often generated by NetworkManager.
  • nmcli con mod NAME ipv4.dns "A B" : Sets persistent DNS servers for a NetworkManager connection profile.
  • nmcli con down NAME && nmcli con up NAME : Re-applies connection settings and regenerates resolver state.
