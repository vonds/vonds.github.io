Diagnose why nginx.service fails to start using systemd status and journal evidence. Resolve the root cause, restart cleanly, verify runtime health, and ensure the service is enabled for future boots.

services troubleshooting core

Scenario

The nginx.service is failing to start on boot. You must investigate the cause, fix it, and ensure it runs now and on future boots.

Operator context

This is a standard production workflow: confirm failure, pull the exact error from the journal, resolve the root cause, then prove it stays healthy after restart and across reboots.

Objective

  • Confirm the service failure using systemctl status.
  • Collect detailed failure evidence using journalctl -xeu.
  • Identify the cause (port conflict) from logs.
  • Confirm what is listening on the conflicting port.
  • Restart the service and confirm it transitions to active (running).
  • Ensure the service starts on boot using systemctl enable.

Concepts

  • systemctl status is triage: it shows unit state and recent context, not the full failure story.
  • The journal is the source of truth for the exact failure line (avoid guessing).
  • A bind failure (Address already in use) means another process already owns the port.
  • Fixing the conflict is separate from restarting the unit. Restart only after the conflict is resolved.
  • enable controls boot-time behavior; it does not start or validate the service right now.

Walkthrough

Step 1: Check the service status.
Command 
systemctl status nginx

This is your initial triage view: loaded state, enablement, active state, and the failing ExecStart context. If it is failing, your next step is the journal for the exact error text. 

# Example output:
● nginx.service - A high performance web server
   Loaded: loaded (/lib/systemd/system/nginx.service; enabled)
   Active: failed (Result: exit-code) since ...
   Process: 1356 ExecStart=/usr/sbin/nginx -g 'daemon on;' (code=exited, status=1/FAILURE)
Step 2: Pull detailed logs for the unit.
Command 
journalctl -xeu nginx

This filters the journal to just the nginx unit while showing relevant recent errors. The goal is to extract the root-cause line, not to guess. 

# Example output:
Jul 18 12:43:01 nginx[1356]: nginx: [emerg] bind() to 0.0.0.0:80 failed (98: Address already in use)
Jul 18 12:43:01 systemd[1]: nginx.service: Failed with result 'exit-code'.
Diagnosis

The bind failure means something else is already listening on port 80. Identify the owner, resolve the conflict, then restart nginx.

Step 3: Identify what is listening on port 80.
Command 
sudo ss -ltnp | grep ':80 '

This confirms which process owns the port. Use that process name and PID as your evidence before you change service state. 

# Example output:
LISTEN 0      511          0.0.0.0:80        0.0.0.0:*    users:(("apache2",pid=912,fd=4))
Step 4: Stop the conflicting service (example).
Command 
sudo systemctl stop apache2

Replace apache2 with the unit you found listening on port 80. The goal is to free the port so nginx can bind successfully.

Step 5: Restart nginx and confirm it is healthy.
Command 
sudo systemctl restart nginx

A restart re-runs the unit’s start sequence. If the root cause is fixed, the unit should transition to active (running).

Verification 
systemctl status nginx
# Example output:
● nginx.service - A high performance web server
   Loaded: loaded (/lib/systemd/system/nginx.service; enabled)
   Active: active (running) since ...
Step 6: Ensure nginx starts on boot.
Command 
sudo systemctl enable nginx

Enabling a unit creates the appropriate symlink in the target’s wants directory (commonly multi-user.target.wants for server services). This makes behavior persist across reboots. 

# Example output:
Created symlink /etc/systemd/system/multi-user.target.wants/nginx.service → /lib/systemd/system/nginx.service

Common breakpoints

Port 80 still in use after stopping the other service

Another process may also be listening, or the service may be socket-activated. Re-check with ss -ltnp and confirm the current owner of port 80.

nginx still fails after freeing the port

The failure mode may have changed. Re-run journalctl -xeu nginx and look for the newest root-cause line (syntax error, missing include, permissions, etc.).

Enable succeeds, but service fails on reboot

Enablement only schedules startup; it does not validate configuration. Check boot-time logs for that boot and confirm dependencies are available.

Service is active but not reachable externally

Verify bind address and firewall rules. A service can be running while only listening on loopback. Use ss -ltnp to confirm the bind interface and port.

Cleanup checklist

This lab changes runtime state if you stop a conflicting service and may change boot behavior if you enable nginx. Confirm nginx is healthy and that port 80 is owned by the expected process.

Commands 
systemctl status nginx
sudo ss -ltnp | grep ':80 '
Success signal

systemctl status nginx shows active (running), and ss shows nginx owning port 80.

Reference

  • systemctl status <unit>: Shows state and recent context for a systemd unit.
    • <unit>: Unit name (example: nginx or nginx.service).
  • journalctl -xeu <unit>: Shows recent journal entries for a specific unit.
    • -u <unit>: Filter output to the specified unit.
    • -e: Jump to the end of the journal.
    • -x: Show explanatory text when available.
  • ss -ltnp: Shows listening TCP sockets and the processes that own them.
    • -l: Show listening sockets only.
    • -t: TCP sockets only.
    • -n: Numeric output (no DNS/service name resolution).
    • -p: Show the process using the socket (requires privileges for full detail).
  • systemctl stop <unit>: Stops a running unit immediately.
    • <unit>: Unit name to stop (example: apache2).
  • systemctl restart <unit>: Stops and then starts the unit.
    • <unit>: Unit name to restart (example: nginx).
  • systemctl enable <unit>: Configures a unit to start automatically at boot.
    • <unit>: Unit name to enable (example: nginx).
