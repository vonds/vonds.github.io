Lab 83: Linux Web-Based Administration (Cockpit)

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Install and enable Cockpit to provide a secure web-based administration dashboard for Linux systems. Verify socket activation and confirm the service is listening on port 9090.

services admin security

Scenario

A team wants to manage Linux systems through a secure web-based dashboard. Your task is to install and configure Cockpit so administrators can access a browser-based interface for system visibility and management.

Operator context

This is a typical ops enablement task: install the management surface, ensure it is reachable only through a controlled port, and prove readiness with both a browser check and a socket-level listener verification.

Objective

  • Install Cockpit using your distro’s package manager.
  • Enable and start Cockpit via cockpit.socket .
  • Open the local Cockpit UI in a browser at https://localhost:9090 .
  • Verify Cockpit is listening on port 9090 using ss .

What You’ll Practice

  • Installing admin tooling via apt , dnf , yum , or pacman .
  • socket activation vs service activation in systemd ( cockpit.socket ).
  • Accessing a local TLS-enabled management UI on a non-default port ( 9090 ).
  • Confirming listener state with ss -tuln .

Walkthrough

Step 1 : Install the Cockpit package.
Command 
# Pick the command that matches your distro:

# Debian/Ubuntu
sudo apt install cockpit -y

# RHEL/CentOS/Fedora
sudo dnf install cockpit -y
# or (older flows)
sudo yum install cockpit -y

# Arch
sudo pacman -S cockpit

Installing Cockpit provides the web UI packages and systemd units needed to expose the management interface. On many systems, Cockpit is designed to be started via a systemd socket, which provides controlled activation and a clean readiness model. 

# Example result (varies by distro):
# cockpit installed successfully.
Step 2 : Enable and start the Cockpit socket.
Command 
sudo systemctl enable --now cockpit.socket

Cockpit is commonly exposed via cockpit.socket rather than running continuously as a long-lived service. Enabling the socket ensures the UI is available and will be activated when a client connects. 

# Quick verification:
systemctl status cockpit.socket
Step 3 : Open Cockpit in your browser.
Command 
# Any of the following are acceptable:
xdg-open https://localhost:9090
# or
firefox https://localhost:9090
# or
chromium https://localhost:9090

Cockpit runs over HTTPS on port 9090 by default. In lab environments, you may see a browser warning about the certificate; the key validation is that you can reach the login page locally. 

# Expected result:
# Cockpit login page loads at https://localhost:9090
Step 4 : Verify Cockpit is listening on port 9090.
Command 
ss -tuln | grep 9090

This is your socket-level proof that the management interface is reachable. You should see a listener on 0.0.0.0:9090 and often on [::]:9090 as well. 

Netid State  Recv-Q Send-Q Local Address:Port   Peer Address:Port
tcp   LISTEN 0      128    0.0.0.0:9090        0.0.0.0:*
tcp   LISTEN 0      128       [::]:9090           [::]:*

Reference

  • cockpit : Web-based administration interface for Linux servers.
  • systemctl enable --now cockpit.socket : Enables Cockpit socket activation and starts it immediately.
  • https://localhost:9090 : Default Cockpit UI endpoint (HTTPS, port 9090).
  • ss -tuln : Lists listening sockets without name resolution.
  • ss -tuln | grep 9090 : Confirms Cockpit is listening on port 9090.
  • xdg-open : Opens a URL using the system’s default browser.
