Lab 128: Networking troubleshooting

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Restore inbound SSH when the service is listening on port 22 but clients still get “port closed.” Verify sshd is listening, confirm firewalld is running, inspect the active zone’s allowed services, and then allow ssh permanently and reload the firewall.

networking firewalld ssh incident troubleshooting

Scenario

A teammate says, “The box is up, but nobody can SSH into it anymore.” You have console access. Your job is to restore inbound SSH quickly.

Constraints

SSH should be running and listening on port 22. The interface and cable are fine. The failure is caused by firewall policy.

Objective

  • Prove sshd is listening on TCP/22.
  • Confirm firewalld is active.
  • Inspect which services are allowed in the active zone.
  • Allow SSH permanently and reload firewall rules.
  • Verify SSH is now permitted by policy.

Mental model

“Port closed” can mean two different things:

  • Service not listening: nothing is bound to the port. Fix the daemon first.
  • Firewall blocking: the service is listening, but the firewall drops/rejects inbound traffic. Fix policy in the active zone.
Fast isolation

First prove the daemon is listening (ss). Then prove the firewall is enforcing policy (firewall-cmd --state) and that SSH is not allowed in the active zone.

Walkthrough

Step 1: Confirm sshd is listening on port 22.
Command 
ss -lntp | grep ':22'

You want to see listeners on 0.0.0.0:22 and/or [::]:22 owned by sshd. That proves the daemon is up and bound correctly.

Step 2: Confirm firewalld is running.
Command 
sudo firewall-cmd --state

If it returns running, policy is being enforced by firewalld.

Step 3: Inspect allowed services in the active zone.
Command 
sudo firewall-cmd --list-services

In this outage, SSH is missing from the allowed services list, so inbound TCP/22 is being blocked.

Step 4: Allow SSH permanently.
Command 
sudo firewall-cmd --permanent --add-service=ssh
Why permanent

Using --permanent updates the saved configuration so the fix survives reloads and reboots. You still need a reload for it to take effect immediately.

Step 5: Reload firewalld to apply changes.
Command 
sudo firewall-cmd --reload

This applies the permanent configuration to the running firewall.

Step 6: Confirm SSH is now allowed.
Command 
sudo firewall-cmd --list-services

You should now see ssh in the services list. At this point, remote SSH attempts should work.

Common breakpoints

sshd not listening

If ss shows no listener on :22, fix the service first: sudo systemctl status sshd and sudo systemctl start sshd.

Wrong zone or interface not assigned

--list-services shows the active zone. If SSH is allowed but still blocked, confirm the interface is in the expected zone with sudo firewall-cmd --get-active-zones.

Port is allowed but still unreachable

Check SELinux, TCP wrappers, cloud security groups, or upstream firewalls. Also confirm the client is actually targeting the correct IP and port.

Need a temporary fix

If you must restore access immediately and do not care about persistence, drop --permanent. In production, you usually do both: immediate fix plus persistent config.

Reference

  • ss -lntp | grep ':22': confirm a service is listening on TCP/22.
  • firewall-cmd --state: confirm firewalld is running.
  • firewall-cmd --list-services: show services allowed in the active zone.
  • firewall-cmd --permanent --add-service=ssh: allow SSH permanently.
  • firewall-cmd --reload: apply permanent config to runtime.
