Lab 117: Debian Package Management with dpkg

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Manage Debian packages directly with dpkg : query package status, list installed packages, enumerate files owned by a package, and identify file ownership. Install a local .deb , resolve dependencies safely with APT, then remove and purge the package to understand the full lifecycle.

packages debian core

Scenario

You are working on a Debian-based system where package work needs to be done at the dpkg level. You must verify package state, determine which package owns a given file, and install a local .deb artifact. When the install leaves the package unconfigured due to dependency issues, you will use APT to correct the system, then remove and fully purge the package.

Operator context

dpkg installs packages but does not resolve dependencies. In real operations, you often pair dpkg with APT to complete the transaction safely and return the system to a consistent state.

Objective

  • Query package status using dpkg -s .
  • List installed packages using dpkg -l .
  • List files installed by a package using dpkg -L .
  • Identify which package owns a path using dpkg -S .
  • Install a local .deb using dpkg -i .
  • Fix dependency issues using apt -f install .
  • Remove a package while keeping config using dpkg -r .
  • Purge a package including config using dpkg -P .

What You’ll Practice

  • Interpreting dpkg status output (installed vs unconfigured).
  • Understanding the difference between “package database operations” and “dependency resolution.”
  • Auditing filesystem ownership by package (useful for incident triage and change tracking).
  • Knowing when to remove vs purge, and how that affects config remnants.

Walkthrough

Step 1 : Show the status of the bash package.
Command 
dpkg -s bash

dpkg -s queries the dpkg database for a package and reports its status. This is how you confirm whether a package is installed and in a healthy “configured” state. 

Package: bash
Status: install ok installed
Priority: required
Version: 5.1-2ubuntu3
Step 2 : List the first 10 installed packages.
Command 
dpkg -l | head -n 10

dpkg -l lists packages known to dpkg and their desired/actual state. This is useful for quick audits and verifying whether a package is installed, removed, or purged. 

Desired=Unknown/Install/Remove/Purge/Hold
||/ Name           Version       Architecture Description
ii  adduser        3.118         all          add and remove users and groups
ii  apt            2.4.5         amd64        commandline package manager
ii  base-files     12ubuntu4     amd64        Debian base system miscellaneous files
ii  bash           5.1-2ubuntu3  amd64        GNU Bourne Again SHell
ii  bsdutils       1:2.37.2-4    amd64        basic utilities from 2.37
ii  coreutils      8.32-4.1      amd64        GNU core utilities
ii  dash           0.5.11+git    amd64        POSIX-compliant shell
Step 3 : List all files installed by the bash package.
Command 
dpkg -L bash

dpkg -L enumerates the paths installed by a package. This is useful for validating expected binaries, locating docs/man pages, and confirming what changes a package introduces to a system. 

/.
 /bin/bash
 /usr/share/doc/bash/README.gz
 /usr/share/man/man1/bash.1.gz
Step 4 : Identify which package owns /bin/bash .
Command 
dpkg -S /bin/bash

dpkg -S maps a path back to its owning package. This is a core troubleshooting tool when you find an unexpected file on disk and need to quickly identify its origin. 

bash: /bin/bash
Step 5 : Install a local .deb file with dpkg.
Note

This step demonstrates a common dpkg outcome: installation succeeds, but configuration fails due to missing dependencies. That is expected when you bypass APT.

Command 
sudo dpkg -i htop_3.0.5-1_amd64.deb

dpkg -i installs a local package file but does not fetch dependencies. If dependencies are missing, dpkg leaves the package “unconfigured” until the system is corrected. 

Selecting previously unselected package htop.
(Reading database ... 123456 files and directories currently installed.)
Preparing to unpack htop_3.0.5-1_amd64.deb ...
Unpacking htop (3.0.5-1) ...
dpkg: dependency problems prevent configuration of htop:
 htop depends on libc6 (>= 2.34); however:
  Package libc6 is not configured yet.
dpkg: error processing package htop (--install):
 dependency problems - leaving unconfigured
Step 6 : Fix missing dependencies after a dpkg install.
Command 
sudo apt -f install

apt -f install attempts to “fix” broken installs by fetching and installing missing dependencies, then completing configuration. This is the standard recovery path when dpkg leaves packages unconfigured. 

Reading package lists... Done
Building dependency tree
Correcting dependencies... Done
The following additional packages will be installed:
  libc6
Setting up libc6 ...
Setting up htop (3.0.5-1) ...
Step 7 : Remove the package but keep configuration files.
Command 
sudo dpkg -r htop

dpkg -r removes the package but preserves configuration files. This is useful when you want the option to reinstall later and keep prior settings. 

(Reading database ... 123460 files and directories currently installed.)
Removing htop (3.0.5-1) ...
Processing triggers for man-db ...
Step 8 : Purge the package including configuration files.
Command 
sudo dpkg -P htop

dpkg -P purges configuration files and removes package traces. Use this when you want a clean removal, or when you are fixing systems that should not retain prior config state. 

(Reading database ... 123450 files and directories currently installed.)
Purging configuration files for htop (3.0.5-1) ...

Reference

  • dpkg -s <pkg> : Shows package status and metadata from the dpkg database.
  • dpkg -l : Lists packages and their state (installed/removed/purged).
  • dpkg -L <pkg> : Lists files installed by a package.
  • dpkg -S <path> : Identifies which installed package owns a path.
  • dpkg -i <file.deb> : Installs a local .deb file (no dependency resolution).
  • apt -f install : Fixes broken installs by resolving dependencies and completing configuration.
  • dpkg -r <pkg> : Removes a package while keeping config files.
  • dpkg -P <pkg> : Purges a package including configuration files.
