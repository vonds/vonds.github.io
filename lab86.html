Lab 86: Configuring an rsyslog Client for Remote Logging

« »
Lab Index
  • Scenario
  • Walkthrough
  • Reference

Configure a Linux host as an rsyslog client that forwards local logs to a central receiver at 192.168.1.10:514. Apply the forwarding rule, restart rsyslog, generate a test message, and confirm processing evidence.

logging networking operations

Scenario

Your organization now has a central rsyslog server listening on 192.168.1.10:514 . Your task is to configure this host as an rsyslog client so local logs are forwarded to the central logging server.

Operator context

Centralized logging only works if clients forward reliably. Your goal is to implement a simple, catch-all forwarding rule, apply it safely, and generate proof that messages are being handled by the logging pipeline.

Objective

  • Install rsyslog on the client system.
  • Create a forwarding rule that sends all facilities/severities to 192.168.1.10 over UDP 514 .
  • Restart rsyslog to apply the configuration.
  • Generate a test message with logger .
  • Verify rsyslog processed the event locally via journalctl .

What You’ll Practice

  • Installing logging components via apt , dnf , or pacman .
  • Rsyslog client forwarding rules using selector syntax ( *.* ).
  • UDP forwarding semantics using a single @ target.
  • Controlled service restart and basic status validation.
  • Generating test events with logger and validating processing via journalctl -u .

Walkthrough

Step 1 : Install rsyslog on the client.
Command 
# Pick the command that matches your distro:

# Debian/Ubuntu
sudo apt install rsyslog -y

# RHEL/Fedora
sudo dnf install rsyslog -y

# Arch
sudo pacman -S rsyslog

The client must have rsyslog installed and running in order to forward system events to a remote receiver. On many distros, rsyslog is already present; installation ensures the unit and config paths exist. 

# Example output (simulated):
rsyslog is already the newest version.
Step 2 : Create a remote forwarding rule.
Command 
# Create a drop-in config for remote forwarding:
sudo nano /etc/rsyslog.d/remote.conf
# or
sudo vim /etc/rsyslog.d/remote.conf

Using a dedicated file under /etc/rsyslog.d/ keeps remote forwarding rules modular and easier to audit. The selector *.* matches all facilities and severities. 

# /etc/rsyslog.d/remote.conf
*.*    @192.168.1.10:514
Note

The single @ indicates UDP forwarding. (A double @@ typically indicates TCP.)

Step 3 : Restart rsyslog to apply the client configuration.
Command 
sudo systemctl restart rsyslog

Restarting applies the new forwarding rule. If rsyslog fails to start, validate your rule syntax and check: systemctl status rsyslog . 

# Example result (simulated):
[ OK ] Restarted rsyslog.service
Step 4 : Generate a test log event.
Command 
logger -t remote-test 'Test message to central rsyslog server'

logger writes to the system logging pipeline. Using a unique tag makes it easy to identify your test message in journald or file-based logs, and to correlate end-to-end forwarding. 

# Expected outcome:
# Test log message generated with tag 'remote-test'.
Step 5 : Confirm rsyslog processed the test message locally.
Command 
sudo journalctl -u rsyslog | tail -n 10

This verifies rsyslog is running and actively processing events. In a real environment, you would also validate receipt on the central server (for example by grepping the receiver logs), but the client-side evidence is your first checkpoint. 

● rsyslog.service - System Logging Service
     Loaded: loaded (/lib/systemd/system/rsyslog.service; enabled; vendor preset: enabled)
     Active: active (running) since Fri 2025-11-28 10:15:42 EST; 2min ago

Nov 28 10:17:35 labhost rsyslogd[1023]: [origin software="rsyslogd"] start
Nov 28 10:17:40 labhost remote-test[1456]: Test message to central rsyslog server
Result

This client is now forwarding logs to 192.168.1.10:514 via UDP.

Reference

  • /etc/rsyslog.d/ : Drop-in configuration directory for rsyslog rules.
  • *.* : Selector matching all facilities and all severities.
  • @host:port : Remote forwarding target using UDP.
  • systemctl restart rsyslog : Restarts rsyslog to apply config changes.
  • logger -t <tag> <message> : Emits a test message into the system logging pipeline.
  • journalctl -u rsyslog : Views rsyslog unit logs via systemd-journald.
  • 192.168.1.10:514 : Central receiver address/port used in this lab scenario.
