Lab 90: Configuring Account Lockout with faillock

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Harden authentication by reviewing system-wide faillock defaults and enforcing a predictable lockout policy after repeated failures. Inspect per-user failure records and safely reset the counter after validating the event is benign.

security authentication hardening

Scenario

Security has reported repeated failed logins against the appuser account. You will review system-wide lockout settings, adjust faillock defaults, and practice checking and resetting failed login counters for appuser .

Operator context

Lockout policy is a balance between reducing brute-force risk and avoiding self-inflicted outages. Your job is to apply a clear policy, validate it, and document the state before resetting counters.

Objective

  • Display system-wide faillock policy values with faillock --system .
  • Edit /etc/security/faillock.conf to enforce a stricter deny/unlock policy.
  • Inspect current failed login attempts for appuser .
  • Reset the failure counter after confirming the event is benign.
  • Re-check the user record and confirm there are no remaining failures.

What Youâ€™ll Practice

  • Understanding lockout primitives: deny , fail_interval , unlock_time .
  • Managing the primary configuration file: /etc/security/faillock.conf .
  • Auditing per-user failure state with faillock --user .
  • Resetting counters safely with faillock --reset after validating intent.

Walkthrough

Step 1 : Check system-wide faillock defaults.
Command 
sudo faillock --system

This displays the active system-wide values that determine how many failures trigger lockout, the window used to count failures, and how long the account remains locked. 

deny = 0
fail_interval = 900
unlock_time = 600
Step 2 : Set a stricter policy in faillock.conf.
Command 
sudo nano /etc/security/faillock.conf
# or
sudo vim /etc/security/faillock.conf

Define a clear deny threshold and unlock time so the system behaves consistently during credential stuffing or brute force. A common baseline is a small deny count and an unlock time long enough to slow attacks without causing a prolonged outage for a legitimate user. 

# Example policy values:
deny = 5
fail_interval = 900
unlock_time = 900
Step 3 : Inspect failed login attempts for appuser.
Command 
sudo faillock --user appuser

This lists recorded authentication failures for the user, including when the attempt occurred and the source address (where available). Use this view to distinguish user error from suspicious activity. 

appuser:
    When                Type  Source             Valid
    2025-01-01 10:15:32 R    192.168.10.50      V
    2025-01-01 10:16:05 R    192.168.10.50      V
    2025-01-01 10:16:40 R    192.168.10.50      V
Step 4 : Reset the failure counter for appuser.
Command 
sudo faillock --user appuser --reset

After you validate the failures are legitimate (typo, scripted health check, or a known internal source), reset the counter to restore normal login behavior. In incident conditions, you typically capture evidence first and reset only after containment decisions. 

# Expected outcome:
Failed login counter for appuser has been reset.
Step 5 : Confirm there are no remaining failures for appuser .
Command 
sudo faillock --user appuser

A final check verifies the reset took effect and ensures the account is not still accumulating failures from an ongoing source. 

appuser:
    No login failures.

Reference

  • faillock --system : Displays current system-wide lockout policy values.
  • /etc/security/faillock.conf : Primary configuration file for faillock defaults.
  • deny : Number of consecutive failures that triggers lockout.
  • fail_interval : Time window (seconds) used to count failures.
  • unlock_time : Time (seconds) before the lockout is lifted.
  • faillock --user <user> : Shows recorded failures for a specific user.
  • faillock --user <user> --reset : Resets failure records for a user after validation.
