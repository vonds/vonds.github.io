Lab 87: Network Troubleshooting with netcat (nc)

« »
Back to Lab Index
  • Scenario

Use netcat (nc) to validate TCP reachability to an internal web app endpoint and prove end-to-end delivery using a temporary listener on the target host. Confirm evidence from both sides so you can separate “network path” failures from “application” failures.

networking troubleshooting tools

Scenario

A branch office reports that the internal web app at 10.10.20.15:8080 is down. You will use netcat ( nc ) to verify connectivity and confirm whether traffic from the branch can reach the application host.

Operator context

When users report an app is “down,” you need fast proof of where the failure lives: DNS, routing, firewall policy, listener state, or the application itself. netcat is a lightweight, low-friction way to validate the path.

Objective

  • Perform a TCP reachability check to 10.10.20.15:8080 with a timeout and verbose output.
  • Start a temporary netcat listener on the app server to observe inbound traffic.
  • Send a test payload from the branch workstation to confirm end-to-end delivery.

What You’ll Practice

  • Using netcat scan mode ( -z ) to test whether a TCP port is reachable.
  • Adding a connection timeout ( -w ) to avoid hanging during reachability tests.
  • Running a listener with verbose output ( -l , -v ) and numeric-only display ( -n ).
  • Proving delivery by piping a payload into nc from a remote host.

Walkthrough

Step 1 : Validate TCP reachability to 10.10.20.15:8080.
Command 
nc -vz -w 3 10.10.20.15 8080

This uses netcat “scan mode” ( -z ) with verbose output ( -v ) and a 3-second timeout ( -w 3 ) to answer a single question: can you establish a TCP connection to the port? 

# Typical patterns you might see:
# succeeded:   10.10.20.15 8080 open
# failed:      timed out / refused / unreachable
Step 2 : Start a temporary listener on the app server.
Command 
sudo nc -lvnp 8080

This binds to TCP port 8080 and prints inbound connections. The goal is to prove whether branch traffic reaches the host at all. Use this as a controlled diagnostic tool and stop it when you’re done. 

# Simulated listener state:
listening on [any] 8080 ...
Step 3 : Send a payload from the branch to confirm end-to-end delivery.
Command 
echo "health-check" | nc 10.10.20.15 8080

Piping a payload into nc makes the client send bytes immediately. On the server-side listener, you should see the inbound connection and the payload. If Step 1 “succeeds” but Step 3 never shows up on the listener, you’re looking at asymmetric routing, NAT, or policy issues between segments. 

# On the app server listener you would see (example):
Connection received on 10.10.20.15 8080
health-check

Reference

  • nc -vz -w <sec> <host> <port> : Tests TCP connectivity to a port using scan mode with a timeout.
    • -z: Zero-I/O scan mode (do not send data).
    • -v: Verbose output.
    • -w: Timeout in seconds.
  • nc -lvnp <port> : Starts a TCP listener for inbound connections.
    • -l: Listen mode.
    • -v: Verbose output.
    • -n: Numeric-only output (no DNS lookups).
    • -p: Local port to bind.
  • echo "msg" | nc <host> <port> : Sends a simple payload to a listening socket.
  • 10.10.20.15:8080 : Target application host and port used in this lab scenario.
