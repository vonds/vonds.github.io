Lab 135: Fix Wrong Static IP on eth0

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Diagnose a cloned host that booted with the wrong static IPv4 address and cannot reach its gateway. Correct the NetworkManager connection using nmcli, apply the change, and verify gateway reachability plus DNS resolution.

networking troubleshooting nmcli

Scenario

This RHEL host was cloned from a template. The eth0 connection came up with the wrong static IP, so the machine cannot reach the gateway. You have local console access only and must fix the NetworkManager connection.

Operator context

Cloned images frequently reuse network profiles. Your goal is to correct the connection profile, apply it cleanly, then prove success with both routing and DNS checks.

Objective

  • Identify the incorrect IPv4 address currently on eth0.
  • Find the NetworkManager connection bound to eth0.
  • Set the exact target static IP and gateway.
  • Set DNS servers to 1.1.1.1 and 8.8.8.8.
  • Bring the connection down and back up to apply changes.
  • Verify the correct address and default route exist.
  • Prove gateway reachability and hostname resolution.

What You’ll Practice

  • Inspecting live interface state with ip.
  • Identifying NetworkManager connection profiles with nmcli.
  • Setting static IPv4, gateway, and DNS with nmcli con mod.
  • Applying changes safely by bouncing the connection.
  • Validating success using routing + DNS evidence.

Walkthrough

Step 1 : Show the current IPv4 address on eth0.
Command 
ip -4 addr show dev eth0
# OR
ip addr show dev eth0
# OR
ip a show dev eth0

The first symptom is usually obvious: the interface is up but the IP is on the wrong subnet, which prevents reaching the expected gateway. 

2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    inet 192.168.1.20/24 brd 192.168.1.255 scope global noprefixroute eth0
       valid_lft forever preferred_lft forever
Step 2 : Identify the NetworkManager connection bound to eth0.
Command 
nmcli -t -f NAME,DEVICE con show
# OR
nmcli con show

You need the connection profile name so you can modify the correct configuration source. The device name alone is not enough. 

System eth0:eth0
lo:lo
Step 3 : Set the correct static IPv4 address and gateway.
Command 
sudo nmcli con mod "System eth0" ipv4.method manual ipv4.addresses 192.168.50.20/24 ipv4.gateway 192.168.50.1

This corrects the address and the default gateway in the connection profile. It is the minimum needed to restore L3 reachability on the intended network.

Step 4 : Set DNS servers for the connection.
Command 
sudo nmcli con mod "System eth0" ipv4.dns "1.1.1.1 8.8.8.8"

Even with the correct IP and gateway, many workflows still fail without DNS (package installs, repo access, API calls). Set DNS explicitly to match the incident target state.

Step 5 : Bring the connection down and back up to apply changes.
Command 
sudo nmcli con down "System eth0" && sudo nmcli con up "System eth0"

Bouncing the connection forces the running network state to match the updated profile and ensures you are validating the actual applied configuration. 

Connection 'System eth0' successfully deactivated (D-Bus active path: /org/freedesktop/NetworkManager/ActiveConnection/3)
Connection 'System eth0' successfully activated (D-Bus active path: /org/freedesktop/NetworkManager/ActiveConnection/4)
Step 6 : Verify eth0 has the correct IP and default route.
Command 
ip -4 addr show dev eth0
# OR
ip route
# OR
ip r

Your success criteria here is evidence: the correct source IP on the interface and a default route pointing at the intended gateway. 

2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    inet 192.168.50.20/24 brd 192.168.50.255 scope global noprefixroute eth0
       valid_lft forever preferred_lft forever

default via 192.168.50.1 dev eth0 proto static metric 100
192.168.50.0/24 dev eth0 proto kernel scope link src 192.168.50.20 metric 100
Step 7 : Prove gateway reachability and DNS resolution.
Command 
ping -c 2 192.168.50.1 && getent hosts example.com
# OR
ping -c 2 192.168.50.1 && getent ahostsv4 example.com

This final check proves the full operator path: local gateway is reachable and name resolution is working. If both succeed, most “network is broken” symptoms disappear immediately.

Reference

  • ip -4 addr show dev IFACE : Displays the IPv4 address currently applied to an interface.
  • nmcli con show : Lists NetworkManager connection profiles and their attached devices.
  • nmcli con mod NAME ... : Updates persistent connection profile settings (static IP, gateway, DNS).
  • nmcli con down NAME && nmcli con up NAME : Applies profile changes to the live network state.
  • ip route : Confirms default gateway and routing table entries.
  • getent hosts NAME : Confirms system-level DNS resolution through NSS.
