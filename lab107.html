Lab 107: Hardware Inventory Basics

« »

Inventory PCI, USB, and block devices using standard CLI tooling, then map a target PCI device to its bound kernel driver. Confirm the driver module is loaded using module inspection paths.

inventory hardware troubleshooting

Scenario

You need a quick, reliable hardware inventory baseline for a Linux host. Your task is to enumerate PCI and USB devices, inspect block devices and mount points, then pick a PCI device and prove which kernel driver is bound to it and whether that driver module is loaded.

Operator context

Establish this inventory baseline before deeper debugging. You cannot troubleshoot drivers, storage, or connectivity if you do not first confirm what the kernel can actually see.

Objective

  • Enumerate PCI devices including numeric vendor and device IDs.
  • Map a PCI device to its bound kernel driver and candidate modules.
  • Enumerate attached USB devices.
  • Inspect block devices and mount points.
  • Confirm the target driver module is loaded.

What You’ll Practice

  • PCI enumeration with lspci -nn for stable device identification.
  • Driver binding inspection with lspci -nnk to distinguish “driver in use” from “available modules.”
  • USB inventory with lsusb .
  • Block device and mount inspection with lsblk .
  • Module load verification with lsmod and /proc/modules .

Walkthrough

Step 1 : List PCI devices including vendor:device numeric IDs.
Command 
lspci -nn

Numeric IDs provide stable identification across vendors and distributions. This helps when correlating devices with driver support matrices or kernel module names. 

00:03.0 Ethernet controller [0200]: Intel Corporation 82540EM Gigabit Ethernet [8086:100e]
Step 2 : Show which kernel driver is in use for PCI devices.
Command 
lspci -nnk

This reveals the active binding state. “Kernel driver in use” is the actual runtime driver, while “Kernel modules” lists candidate modules that could handle the device. 

Kernel driver in use: e1000
Kernel modules: e1000
Step 3 : List attached USB devices on the system.
Command 
lsusb

USB inventory is useful for validating keyboards, storage adapters, wireless receivers, and other peripherals that may not show up in PCI output. 

Bus 001 Device 003: ID 046d:c534 Logitech, Inc. Unifying Receiver
Step 4 : Show block devices with size and mountpoints columns.
Command 
lsblk -o NAME,MAJ:MIN,RM,SIZE,RO,TYPE,MOUNTPOINTS

lsblk provides a clean view of disks, partitions, and mountpoints. This helps you confirm what is actually mounted and where before you touch storage. 

sda
├─sda1 /boot
└─sda2 /
Step 5 : Verify that the Ethernet driver’s kernel module is currently loaded.
Command 
lsmod | grep -E '^e1000(\s|$)'

This confirms the driver module is loaded in the running kernel. If the module is not present, the driver may be built-in, blacklisted, missing, or not bound to the device. 

e1000               155648  0
Alternate 
grep e1000 /proc/modules

/proc/modules is another direct view of currently loaded modules. It can be useful in minimal environments where output parsing is preferred.

Reference

  • lspci -nn : Lists PCI devices and includes numeric vendor and device IDs.
    • -n : Displays numeric IDs; typically used as -nn to show both text and numeric IDs.
  • lspci -nnk : Shows kernel driver binding and candidate modules.
    • -k : Displays the kernel driver currently bound to each device.
  • lsusb : Lists USB buses and attached devices.
  • lsblk -o ... : Lists block devices with custom columns like size and mountpoints.
    • MOUNTPOINTS : Shows all mountpoints associated with a device.
  • lsmod : Lists currently loaded kernel modules.
  • /proc/modules : Kernel-exported view of loaded modules (name, size, use count).
