Lab 127: Networking troubleshooting

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Fix a Linux VM that can reach the local gateway but cannot reach the internet. Diagnose in a clean order: interface addressing, local reachability, routing to public IPs, then DNS. Restore connectivity by adding a missing default route and correcting systemd-resolved DNS with resolvectl.

networking routing dns troubleshooting systemd-resolved

Scenario

You SSH into a Linux VM (net-ops-127) after an app deploy. The node can reach the local gateway (192.168.56.1) but cannot reach the internet. Your job is to restore outbound connectivity and ensure DNS works again.

Constraints

Interface: eth0. Gateway should be 192.168.56.1. DNS should end up using 1.1.1.1 and 8.8.8.8. Resolver stack is systemd-resolved, so use resolvectl.

Objective

  • Confirm an IPv4 address is present on eth0.
  • Validate local reachability to the gateway.
  • Differentiate routing failures from DNS failures.
  • Detect a missing default route and add it correctly.
  • Use resolvectl to inspect and repair DNS server configuration.
  • Verify success with both IP and hostname tests.

Mental model

“No internet” is not one problem. It is usually one of these layers:

  • Addressing: the interface has no IP, or the IP/subnet is wrong.
  • Local path: you cannot reach the gateway (L2/L3 local issue).
  • Routing: you can reach the gateway, but there is no default route to the internet.
  • DNS: you can reach public IPs, but hostnames fail to resolve.
Order matters

Always test public IP reachability before you chase DNS. If ping 1.1.1.1 fails, DNS is not the root cause yet.

Walkthrough

Step 1: Confirm IPv4 addressing on eth0.
Command 
ip -4 addr show eth0

You want a valid inet address and the correct prefix (here, /24) before you go further.

Step 2: Prove the gateway is reachable.
Command 
ping -c 2 192.168.56.1

This confirms local L2/L3 connectivity. If this fails, fix the local network first (cable, VLAN, NIC, IP/subnet, firewall).

Step 3: Test routing with a public IP (bypass DNS).
Command 
ping -c 2 1.1.1.1

If this fails while the gateway ping succeeds, the problem is usually routing (often a missing default route) or an upstream firewall/NAT issue.

Step 4: Inspect the routing table.
Command 
ip route

You should see a line starting with default via .... If it is missing, the host has no path to networks outside its local subnet.

Step 5: Add the missing default route.
Command 
sudo ip route add default via 192.168.56.1 dev eth0
Note

This fixes routing immediately but may not persist across reboot depending on how networking is managed (NetworkManager, systemd-networkd, etc.). The lab focuses on incident-style restore first.

Step 6: Verify the default route exists.
Command 
ip route

Confirm you now see default via 192.168.56.1 dev eth0.

Step 7: Re-test public IP reachability.
Command 
ping -c 2 1.1.1.1

At this point, public IP pings should work. Now you can move to DNS.

Step 8: Test DNS resolution with a hostname.
Command 
ping -c 1 example.com

If you see a name resolution error, DNS is misconfigured (or blocked) even if routing is now correct.

Step 9: Inspect resolver configuration (systemd-resolved).
Command 
resolvectl status

Look at the DNS servers assigned to eth0. In this lab, a bad resolver is configured (192.168.56.250).

Step 10: Set working DNS servers on eth0.
Command 
sudo resolvectl dns eth0 1.1.1.1 8.8.8.8

This updates the link-scoped DNS configuration for eth0.

Step 11: Validate name resolution using getent.
Command 
getent hosts example.com

getent confirms NSS resolution is working. This is a clean validation step before you rely on application behavior.

Step 12: Final check with a hostname ping.
Command 
ping -c 1 example.com

If this works, outbound connectivity is restored and DNS is functioning.

Common breakpoints

Gateway ping fails

Check the interface state, IP/prefix, and L2 connectivity. Confirm the correct interface name (ip link) and verify local subnet settings.

Public IP ping fails even with a default route

The upstream device may not be performing NAT, a firewall may block egress ICMP, or the VM network may be isolated. In real systems, follow up with traceroute and policy checks.

DNS still fails after resolvectl dns

Confirm systemd-resolved is running, confirm the link-scoped DNS shows under Link (eth0), and retest with resolvectl query example.com. Also consider firewall rules for UDP/TCP 53.

Route change disappears later

ip route add is runtime state. For persistence, set it in the network manager used by the system (NetworkManager, systemd-networkd, or distro config files).

Reference

  • ip -4 addr show eth0: confirm IPv4 address and prefix.
  • ip route: inspect routing table and default route.
  • ip route add default via <gw> dev <if>: restore outbound routing.
  • ping -c N <ip>: test reachability (use an IP to bypass DNS).
  • resolvectl status: inspect systemd-resolved resolver state.
  • resolvectl dns eth0 1.1.1.1 8.8.8.8: set DNS servers for a link.
  • getent hosts example.com: validate NSS name resolution.
