Lab 133: Investigate Port 443 Failure (Path + Firewall)

« »
Back to Lab Index
  • Scenario
  • Walkthrough
  • Reference

Troubleshoot why HTTPS works from other servers but fails from this host. Prove DNS and ICMP are healthy, isolate the failure to TCP/443, fix the local firewall policy, and verify HTTPS connectivity.

networking troubleshooting firewall

Scenario

From this server only, https://checkout.lab.example fails. Other servers can reach it. Your job is to isolate the failing layer and fix local policy so outbound HTTPS works.

Operator context

Do not guess. Start at name resolution, then path, then port. If ICMP works but TCP/443 times out, treat it as a policy or state issue on this host until proven otherwise.

Objective

  • Confirm the hostname resolves to an IP.
  • Confirm basic reachability to the IP.
  • Prove TCP/443 is failing from this host.
  • Inspect the active firewall zone and allowed services.
  • Allow HTTPS and apply the change.
  • Verify HTTPS connectivity succeeds.

What You’ll Practice

  • DNS verification with getent or dig.
  • Path sanity checks using ping.
  • Testing a TCP port with nc.
  • Auditing firewalld zone state and services.
  • Applying a permanent firewall change and validating it.

Walkthrough

Step 1 : Confirm the hostname resolves to an IP.
Command 
getent hosts checkout.lab.example
# OR
dig +short checkout.lab.example

If resolution fails, stop here and fix DNS. If it resolves, you now have a concrete target IP to test without mixing layers. 

203.0.113.20   checkout.lab.example
# OR
203.0.113.20
Step 2 : Test basic reachability to the resolved IP.
Command 
ping -c 2 203.0.113.20

This confirms you can reach the target network path at a basic level. ICMP success does not prove TCP will work, but it narrows the problem space. 

PING 203.0.113.20 (203.0.113.20) 56(84) bytes of data.
64 bytes from 203.0.113.20: icmp_seq=1 ttl=52 time=18.4 ms
64 bytes from 203.0.113.20: icmp_seq=2 ttl=52 time=18.1 ms

--- ping statistics ---
2 packets transmitted, 2 received, 0% packet loss
Step 3 : Test TCP connectivity to port 443 (the failing layer).
Command 
nc -vz checkout.lab.example 443
# OR
nc -vz 203.0.113.20 443

A timeout here (while ping works) points to a policy issue: local firewall, routing policy, or upstream filtering scoped to this host. 

nc: connect to checkout.lab.example (203.0.113.20) port 443 (tcp) failed: Connection timed out
Step 4 : Check the active firewalld zone.
Command 
sudo firewall-cmd --get-active-zones

You need the active zone before you can reason about policy. Fixing the wrong zone is a common failure mode during incident response. 

public
  interfaces: eth0
Step 5 : List services allowed in the active zone.
Command 
sudo firewall-cmd --zone=public --list-services

If https is missing, outbound or forwarded HTTPS traffic may be blocked depending on how the host is configured and what policy rules apply. 

ssh dhcpv6-client
Step 6 : Allow HTTPS permanently in the public zone.
Command 
sudo firewall-cmd --zone=public --add-service=https --permanent

This adds the policy change to the permanent configuration. You still need a reload to apply it to the runtime ruleset. 

success
Step 7 : Reload firewalld to apply permanent changes.
Command 
sudo firewall-cmd --reload

Reload updates the runtime ruleset to match the permanent configuration. 

success
Step 8 : Verify HTTPS connectivity succeeds.
Command 
curl -I https://checkout.lab.example
# OR
nc -vz checkout.lab.example 443

This is your closure check. The goal is not “firewall rules changed,” it’s “HTTPS works again from this host.” 

HTTP/2 200
server: nginx
content-type: text/html
# OR
Connection to checkout.lab.example 443 port [tcp/https] succeeded!

Reference

  • getent hosts NAME : Resolves names through NSS using the system-configured order.
  • dig +short NAME : Fast DNS lookup that returns only the answer section.
  • ping -c N TARGET : Tests basic IP reachability (ICMP).
  • nc -vz HOST PORT : Tests whether a TCP port is reachable.
  • firewall-cmd --get-active-zones : Shows active zones and their bound interfaces.
  • firewall-cmd --list-services : Lists services allowed in a zone.
  • firewall-cmd --add-service=https --permanent : Adds HTTPS service allowance to persistent configuration.
  • firewall-cmd --reload : Reloads runtime firewall rules from permanent config.
  • curl -I URL : Fetches HTTP headers to validate HTTPS connectivity.
